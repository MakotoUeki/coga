イントロダクション

このガイドは、Webサイトやアプリケーションを認知障害や学習障害を持つ人々にとってフレンドリーにすることをアシストします。このガイドにおけるパターンは、デザインや設計プロセスのアクセシビリティを向上させるための実践的なガイダンスとなっています。

ここで紹介する「目標」と「パターン」は、The Web Content Accessibility Guidelines WCAG [[WCAG22]] の達成基準を超えた補足ガイダンスを提供するものです。これらは、規定であるWCAG [[WCAG22]] に含まれていないために対処されないバリアにも対処することを目的としています。

このガイドは、デザインの「目標」に分けられています。これらの「目標」の概要は、「サマリー」のセクションにも記載されています。「目標」と「関連するユーザーストーリー」を理解するだけでも、認知障害や学習障害を持つ一部のユーザーがコンテンツをより利用しやすくするにあたって、デザイナーの皆さんの参考になるでしょう。

それぞれの「目標」には、ユーザーのニーズに応えるために何をすべきかを説明する、いくつかの実践的なパターン（コントロールやその他の要素のための繰り返しのデザイン）が含まれています。これらのパターンを実装することは、認知障害や学習障害がある人たちが自力でコンテンツを利用できるようにするために必要不可欠です。

目標 1: それが何なのか、どのように使用するかをユーザーが理解できるようにする。

認知障害や学習障害があるユーザーは、方向感覚や学習に問題がある場合があります。これは、ユーザーがサイト内で方向感覚を失ってしまうことを意味します。

新しいことを学んだり、新しい情報を覚えたりすることは、認知障害や学習障害がある人にとって特に難しいことです。また、新しいデザインパターンを学ぶのに苦労したり、学べなかったりすることもあります。コントロール、アイコン、要素をシンプルで馴染みがあるものにすることで、助けになります。

それが何であり、どのように使用するかをユーザーに明確に示してください。これには、以下の目的を明確に示すことが含まれます：

  • ウェブサイト
  • サイト内のセクション
  • ウェブページ
  • ページ内のセクション
  • （フォーム）コントロール

ヘッダー、ラベル、その他の道しるべを使用して、ユーザーがページ、領域、またはコントロールの目的を知ることができるようにする。

各ページのコントロールや要素の使用方法をユーザーが理解できるようにする。

新しいデザインを覚えるのに苦労するユーザーのために、馴染みのあるデザイン パターン、用語、およびアイコンを使用する。コントロールやその他の要素の外観、位置、およびインタラクションが、サイト全体で慣れ親しんだものであり、一貫していることを確認する。

コントロールとそれが作用するコンテンツとの間に明確な関係を示すことで、実行しうるアクションにより何が起こるかをユーザーが理解し、潜在的な混乱を軽減する。

ページの目的を明確にする ［パターン］

ユーザーニーズ

ページの文脈や目的を知りたい。

関連するユーザーストーリー： 3.1.1 明確な目的

やるべきこと

ユーザーがコンテンツの目的を知ることができるようにするために、以下を用いる：

  • ページの目的を要約した明確なタイトルや見出し、または
  • その他、認知障害や学習障害を持つユーザーによってテストされた明確な道しるべ。
メリット

記憶力や注意力が低下している人はもちろん、加齢に伴う物忘れやAD(H)Dなどで気が散りやすい人など、多くの人に役立ちます。

例えば、軽度の認知症の方がオンラインショッピングを利用するとします。気が散ってしまい、再び画面を見たときには何をしていたか忘れてしまっているのです。各ページの上部に明確な見出しをつけることによって、そのページが何についてのページなのか、何をしているのかを明確に示します。

別の例では、AD(H)Dのユーザーが動画で情報を探しています。彼らは動画のタイトルから、この動画には自分が必要としている情報があるとわかります。

詳細

見出しは、特定のページの目的を明確にするものです。

可能であれば、ユーザーがどのようにしてそのページにたどり着いたかを理解できるような情報も提供します。例：代表的なナビゲーションでのパンくずリストを明示する、現在選択されているタブをハイライトする、など。

事例

用いる：

  1. ユーザーがどこにいるのかを示すページの見出し

避ける：

  1. ユーザーがどこにいるのかを示す明確な見出しや道しるべがないページ。例えば：
    • 「サービスはご利用いただけません」というページの見出しなど、ユーザーにどこにいるのかを知らせないページの見出し。ユーザーはそのサービスが何に関連しているかを覚えておかなければならない。
    • フォーム内でのステップを明確にしないページの見出し

馴染みのある階層とデザインを用いる［パターン］

ユーザーニーズ

自分の選択肢と実行できるタスクを理解する必要があり、アクションを完了するために操作できるコントロールを特定したい。（新しいインターフェース・デザイン・パターンを学ぶのは難しい）。

関連するユーザーストーリー： 3.1.2 明確な操作

やるべきこと

以下のように、多くのユーザーに馴染みがある一般的なデザインを使用する：

  • デザイン要素
  • アフォーダンス（コントロールの方法を示す視覚的なヒント）
  • パターン
  • レイアウトと視覚的な階層（重要度の順番を示すための要素の配置）
メリット

多くのユーザーは、新しいデザインのメタファーを簡単に学んだり覚えたりすることができません。そういったスキルがない場合、インタラクションしたいアイテムを探すことや、インタラクションによって何ができるのかを知ることが非常に難しく、不可能になることもあいます。ユーザーは迷ったり、圧倒されたりしてしまいます。

ユーザーは、多くのサイトで長期間にわたって頻繁に繰り返されている、よくあるデザイン要素を見つけると、認識できる可能性が高いです。

例えば、軽度の認知障害や認知症のユーザーが、商品を購入するためにサイトにアクセスしたとします。しかし、欲しい商品の代金を支払う場所が見つかりません。そのサイトでは買い物ができず、ただ情報を提供しているだけだと思ってしまいます。

詳細

よくあるデザイン要素、アフォーダンス、パターンには、以下のようなものがあります：

  • 標準的なレイアウトと視覚的な階層構造（重要性の順序を示す要素の配置）。ユーザーが期待する場所に要素を配置する。例えば、英語のサイトでは：
    • Webサイトで右上にある検索
    • 左上にあるホームページへのリンク
      • トップナビゲーションエリアの「contact us」へのリンク
    • フッターエリアのサイトマップへのリンク
    • フォームの下部にある［送信］ボタン
  • よくあるデザインパターン（コントロールなどの要素で繰り返されるデザイン）には、以下のようなものがあります：
    • WAI-ARIAオーサリングのベストプラクティス [[wai-aria-practices-1.2]],
    • 最も人気のあるサイトで使用されているパターン
    • 非常に一般的なナビゲーションのデザインパターンと共通のアイコン
    • ナビゲーション機構とアイコンのためのプラットフォーム固有（オペレーティングシステム）のユーザー インターフェイス デザイン
    • パーソナライズ可能なアダプティブなユーザー インターフェイス デザイン (上記参照)
  • 以前のバージョンのユーザー インターフェイス (デザイン)。ユーザーが慣れ親しんだアプリケーションの以前のバージョンに戻せるようにする
  • リンクのように見えるリンクと、ボタンのように見えて動作するボタン。例えば：
    • ページ全体で標準的なスタイルを用いてリンクに下線を引く。
    • リンクは一般的には新しいページにナビゲートする。
    • ボタンは一般的にはアクションを実行する。
まずはじめに
  1. Webページをデザインする際には、ユーザーが期待するような外観や動作をする標準的なコンポーネントを選択する。また、次のような標準的なレイアウトを用いる：
    • 左上にHOMEページへのリンク
    • 上部にナビゲーション
    • 右上に検索
  2. ページ内に明快な視覚的階層を作る
事例

用いる：

  1. ウェブページでよくあるレイアウト

避ける：

  1. 馴染みのないレイアウト
  2. 馴染みのない視覚的な構造

視覚的なデザインに一貫性を持たせる ［パターン］

ユーザーニーズ

自分の選択肢と実行できるタスクを理解する必要があり、アクションを完了するために操作できるコントロールを特定したい。

関連するユーザーストーリー： 3.1.2 明確な操作

やるべきこと

ページのグループの中でビジュアルデザインに一貫性を持たせます。

メリット

多くのユーザーは、新しいデザインを学び、要素を認識するのに長い時間を要します。一度覚えさせた要素は、サイト全体で使用する必要があります。

例えば、加齢に伴う物忘れがある年配のユーザーは、新しいデザインを覚えるのに時間がかかります。彼らがサイトに来たとき、最初のページを理解するのには時間がかかりますが、その後、次のページで何をすべきかがわかります。次のページが最初のページと違っていて、しかも覚えるのが難しいと、疲れてしまい、ミスが増えてしまいます。そして、3つ目の難しいページになると、認知的な負荷が大きくなり、タスクを完了できなくなります。

詳細

これには以下が含まれます：

  • 見出しのスタイル、フォントの選択、アイコン、色、コントロールの視覚的な外観、ボタンやリンクを含む、一貫性のあるデザインテーマ。
  • 同じ見出しレベルの見出しは、同じフォントと視覚的なスタイルになっている。
  • 同じ機能と役割を持つアイコン、コントロール、およびメニュー項目は、同じ外観とスタイルになっている。
  • 類似した機能と役割を持つ要素の状態とフォーカスは、サイト全体で同じスタイルになっている。
  • レイアウトがコンテンツの複数のブロックを通じて一貫している。例えば、インタラクティブな要素やナビゲーション コントロールの位置。
  • コンテンツの構造と情報提示のスタイルが一貫している。例えば、テキスト、アイコン、画像、および箇条書きのブロックの構成。
まずはじめに

コンテンツを追加する前に、情報のデザインを設計する。色やフォントの選択、テキストや画像を配置するエリアなどを検討する。

事例

用いる：

  1. サイト全体で同じように見える同じレベルの見出し
  2. サイト全体で統一されたコントロールの外観。例えば：
    • ウェブサイトにある 2 つの送信ボタンは、どちらも同じように見え、同じように機能する。
    • ウェブサイト上のすべての選択されたラジオボタンは、同じように見える。
    • あるアイテムにフォーカスが移動すると、そのアイテムはフォーカスを受け取り、実行できるようになる。全てのリンクのキーボード フォーカス インジケーター（どの要素がフォーカスされているかを示すアウトライン）は同じに見える。
  3. 共通の機能のための一貫した位置。例えば：
    • 検索ボックスがサイト全体で常に同じ場所にある。

避ける：

  1. 似たような機能を持つ要素が異なって見えること。例えば：
    • 1つのページに6つの見出しレベル 2がある。4つはTimes New Roman、2つはHelveticaを使用してスタイリングされている。
  2. 似たような機能を持つ要素が異なる場所に表示されている。例えば：
    • 3つのページには送信ボタンがある。1つはフォームの上部に、1つはフォームの下部にそしてもう1つはサイドバーに配置されている。

各ステップを明確にする［パターン］

ユーザーニーズ

気が散った後でも、Web サイト、アプリケーション、またはマルチステップ プロセスのアーキテクチャの中で自分がどこにいるかを認識したい。

関連するユーザーストーリー： 3.1.1 明確な目的

やるべきこと

ユーザーがサイトやタスクの中での現在位置を確認できるように、パンくずリストや「ここにたどり着いた経緯（"how I got here"）」ボタン、見出しなどを用意します。

複数のステップがあるプロセスでは、以下のようなものを示します：

  • 完了したステップ
  • 現在のステップ
  • 保留中のステップ
  • すべての重要な選択
メリット

このパターンは、集中力を失ったり、何をしているのか忘れてしまったり、気が散ってしまったユーザーが、現在のアクティビティでの現在位置を確認するのに役立ちます。現在の位置と進捗状況を明確に示すことで、ユーザーが集中力を失った後、大量のコンテンツを読んだり、再起動したりすることなく、継続することができます。

タスクを完了するために必要なステップについての情報を提供することは、ユーザーがそのタスクを正常に完了できるかどうかを判断するのに役立ちます。これは、プロセスを完了するのが難しいと感じることが多いユーザーにとって、特に重要なことです。

以下のような例が挙げられます。

  • 初期の認知症の人は、作業を中断したり、集中力を失ったりすると、自分が何をしていたかを思い出せなくなる。パンくずリストを見ることで、自分がどこにいるのかを思い出し、作業を続けることができる。
  • 注意障害のある人は、注意散漫になったときには、どこで中断してしまったのかを把握する必要がある。
  • 処理能力に問題のある人は、アプリケーションにあまりにも多くのステップがあるかどうか、自分が処理できるかどうかについて、確信が持てない。自分が途中まで進んでいるのを見ることで、プロセス全体に対処できるかどうかを判断できる。
事例

用いる：

  1. プロセス内での現在のステップ、重要な選択、および過去と未来のステップを示すパンくずリスト
  2. ユーザーがコンテンツ内のどの位置にいるのかを明確にする見出し
  3. 「How I got here」ボタン。ユーザーがこのボタンをクリックすると、以下のような現在位置に関する情報が表示されます：
    • そのページ
    • どのようにしてここに来たのか
    • コンテンツ内のどこにいるか

避ける：

  1. 見つけにくく、チェックしにくい、完了したステップや選択肢。例えば：
    • 完了したステップを前のページに配置する
    • コンテンツを隠したり見せたりするコントロール（アコーディオンなど）の中に、選択されたものを配置する

コントロールとその使い方を明確に示す［パターン］

ユーザーニーズ

自分の選択肢と実行できるタスクを理解する必要があり、アクションを完了するために操作できるコントロールを特定したい。

関連するユーザーストーリー： 3.1.2 明確な操作

やるべきこと

コントロールには、明確でわかりやすいデザインを使用してください。どの要素がコントロールなのか、どのように使用するのかを明確にしてください。

以下のような例が挙げられます

  • コントロールに共通のスタイルを使用する（たとえば、リンクに下線を引くなど）。
  • リンクとコントロールに共通のデザインパターンを使用する （たとえば、リンクをクリックするとそのページに移動する）。
  • コントロールの境界線を明確にする。テキスト内のリンクは、適切に識別されていれば、境界線は必要ない（例えば、ヘルプ アイコンには境界線がある）。
  • ユーザーが隣の項目ではなく、コントロールをクリックできるように、コントロールを十分に大きくする。
  • クリックできないアイテムがリンクやコントロールのように見えないようにする。

それができない場合は、コントロールの使用方法を説明した説明書を提供してください。説明書は同じページかワンクリックで表示され、理解しやすい言葉で書かれている必要があります。

メリット

コントロールとは、リンク、ボタン、チェックボックスなど、何かをするためのWebページのパーツです。コントロールに共通のスタイルやデザインパターンがあると、認識しやすく、使い方を理解しやすくなります。

これらのコントロールの目的は、誰かに使ってもらうことです。ユーザーがコントロールを発見したり、使用方法を見つけ出したりする必要があるとすぐに、一部のユーザーは失敗してしまいます。

例えば、加齢に伴う物忘れがある高齢のユーザーは、新しいデザインを学ぶのに時間がかかります。彼らが「sale」という枠で囲まれた見出しがある e コマース サイトにアクセスしたとします。そして、「カートに入れる」ボタンなどのコントロールにはシンプルな大きな文字が使われています。ユーザーは見出しをクリックし、「カートに入れる」ボタン（テキストのように見える）をクリックしません。何度か失敗しているうちに、自分ではコントロールできないと判断し、サイトを離れてしまいます。

一部のユーザーは、コントロールの外観、色、または形状が以前に使用したものと異なる場合、問題を抱えます。例えば、リンクに下線がなく、青や紫のテキストがある場合、一部のユーザーはリンクがあることに気付きません （たとえフォーカスされたときに下線が表示されたとしても）。

記憶力に問題がある場合、独自性があるコントロールを使用することが難しくなります。ページ上のコントロールを見つけるのに時間がかかるかもしれません。似たようなものと少しだけ違う動作をしたとしても、ユーザーによっては毎回使い方を覚え直さなければならないかもしれません。

典型的なコントロールをページ上で使用することで、ユーザーがその使い方を知る手助けになります。より独自性があるコントロールを使用する場合は、わかりやすい説明を盛り込み、簡単に見つけられるようにしてください。ユーザーがどのようにページを使用するか （視覚、聴覚、音声入力） にかかわらず、コントロールを識別し、理解し、使用することが容易でなければなりません。

まずはじめに

標準的なコントロールやデザインパターンを使用する。

新しいコントロールを設計する場合は、以下に挙げることが簡単にできるようにする：

  • 識別する （そこにあることが分かる）
  • 理解する （何をするか分かる）
  • 使用する（どのように使うか分かる）

シンプルなスタイルを採用するか、使い方を説明するわかりやすい説明書を用意してください。さまざまな認知障害や学習障害を持つ人たちと一緒にテストしましょう。

事例

用いる：

  1. 標準的な青と紫のリンクカラー、または下線とはっきりしたテキストカラーを持つリンク（使用されるリンクテキストカラーは、サイト上の静的なテキストには使用されない）。
  2. 標準的なボタンのデザイン（3D の長方形） で、標準的なユーザーのアクションをサポートし、押したときにはっきりとわかるボタン。
  3. タブのように見えて、選択されたときにはそれが明確に分かるタブ。

避ける：

  1. 下線のないリンク。
  2. フォーカス状態が明確なリンクであっても、サイト内の静的なテキストと同じ色のリンク。

コントロールとそれが影響を与えるコンテンツとの関係性を明確にする［パターン］

ユーザーニーズ

すべてのコントロールの使用方法と、各アクションの効果を知りたい。

関連するユーザーストーリー： 3.1.2明確な操作

やるべきこと

コントロールと影響を受けるコンテンツの関係は完全に明確にして、曖昧にしない。

次のようにして達成できます：

  • コントロールを、関連するコンテンツと一緒に視覚的にグループ化する。
  • 影響を与える領域内にコントロールを含める。
  • 別のコントロールやスクロール バーがあるページの領域間に、明確な仕切りや空白を使用する。
  • 複数または入れ子になったスクロール領域を避ける。
メリット

ページ上のコントロールがページの一部でしか動作しない場合、そのコントロールが何に影響を与え、何に影響を与えないのかを見分けるのが難しいことがあります。ユーザーは間違ったコントロールを試すかもしれません。多くのユーザーは、もう一度試してみて、正しいコントロールまたはスクロールバーを発見するでしょう。しかし、認知障害や学習障害を持つ多くのユーザーは、何をすべきかを理解できないかもしれません。他の人は、認知的な過負荷を感じ、その結果、停止してしまいます。アプリケーションが壊れていると思い込んだり、自分には複雑すぎると思い込んだりするかもしれません。このようなユーザーにとっては、アプリケーションは使い物になりません。

ページ上の明確な境界線とグループ化は、コントロールがどの要素に影響を与えるかを示すのに役立ちます。コントロールと関連するセクションの周りにボーダーやその他の視覚的な手がかりがあると、より理解しやすくなります。認知障害や学習障害を持つユーザーが、すべてのページの関係を明確に感じ、コントロールの使い方をすぐに理解できるかどうか、ユーザーテストで確認してください。コントロールが、影響を与える領域内に存在しない場合には、テストは不可欠です。

例えば、認知症のユーザーが、埋め込まれたスクロール可能な領域に使用するスクロールバーを見つけようとしているとします。間違ったスクロールバーを試すと、望んだ効果が得られず、コンテンツが消えたように見えることがあります。

詳細
  • インタラクティブ要素を分ける　例えば、2本のスクロールバーを近接して配置しない。ユーザーによっては、コンテンツの特定のセクションでどちらを使うべきか判断するのが難しいと感じるかもしれません。その代わりに：
    • 視覚的レイアウトとスクロールバーの配置を明確にする。
    • または、コンテンツを2つのページに分ける。
    • または、ページ内の不要な情報を削除することを検討する。
  • 要素とコントロールを関連付ける　スクロールバーやボタンなどのインタラクティブな要素は、影響を与えるコンテンツの近くに配置します。インタラクティブな要素は、それが適用されないコンテンツから離してください。これにより、どの要素がコンテンツの各セクションに影響を与えるのかを簡単に特定することができます。
  • 区切りを使う　明確な区切りの例としては、ハイコントラストのボーダーやホワイトスペースがあります。コントラストが十分に強ければ、背景色の変更も明確な区切りになります。
  • ユーザーが正しいコントロールを見るのを助ける　大きくて分かりやすいボタンやスクロールバーを使用してください。
事例

用いる：

  1. コントロールは、それがコントロールするセクションと明確に関連付けられている。コントロールをセクション内に配置し、セクションの周りには明確な境界線を設ける。セクションの見出しに合わせてコントロールにラベルを付ける。例えば：
    • 図書館のサイトでは、サイト全体を検索する検索ボックスがサイトのメインナビゲーションの右上にある。もう一つの検索ボックスは、図書館のカタログを検索するもの。この検索ボックスは、透明なボーダー、異なる背景色、「Library Catalog」という見出しを持つセクションの中にある。検索実行ボタンには「search catalog」と書かれている。
    • ページ内のあるセクションにスクロールバーが必要で、スクロールバーはセクションの中にあるように見える。

避ける：

  1. 複数のスクロール可能なセクションで、どのスクロールバーをどのセクションで使用するかが不明瞭である。
  2. ページ全体または別のセクションに影響を与える可能性があるような、あるセクションのためのコントロール。例えば：ページのある1つのエリアに関する検索ボックスは、その他のエリアには関係ない。しかし、どのエリアのための検索なのかが不明瞭。
  3. 複数の入れ子になったスクロール領域。

やるべきこと

コントロールやセクションの見出しなどの重要なコンテンツには、馴染みのあるアイコンや画像、シンボルを追加してください。それぞれのアイコンやシンボルは単一の意味を持ち、関連するコンテンツの隣に配置する必要があります。

メリット

言語理解、学習、読解に困難を抱える人は、コンテンツを理解し、必要なコンテンツにナビゲートするためにシンボルを利用することがあります。シンボルは、言語や注意力に困難を抱える人々が、メディアを含むコンテンツをナビゲートするのにも役立ちます。

例えば、失語症の人は、概念を理解する知的能力はありますが、言語が苦手なため、シンボルを頼りにしてページを閲覧し、情報を得ることができます。

また、文字が密集して雑然としたページを画面上で読むのが困難な高齢者にも有効です。テキストコンテンツへの道しるべとなる明確なシンボル、アイコン、画像は、非常に役に立ちます。

詳細
  • 見やすく、拡大しやすい、明確で曖昧さのないアイコンやシンボルを使用する。
  • 文化の違いを意識する。
  • 左から右に記述する言語では、数個のアイコンやシンボルをページに追加する場合、画像をテキストの左側に配置する。
  • 段落やテキストのセクションに複数のシンボルを追加する場合は、シンボルをテキストの上に配置する。
  • [[personalization-semantics-1.0]] のようなパーソナライズセマンティクスを使用して、ユーザーが馴染みのあるシンボルを読み込めるようにする。
まずはじめに

ユーザーが理解しやすい共通のアイコンを使用する。

重要なテキスト、見出し、メディアセクション、「お問い合わせ」ボタン、「ヘルプ」ボタンの横に共通のアイコンを配置する。

事例

用いる：

  1. コンテンツに関連したアイコンで始まる説明や重要なリスト（箇条書き）。例えば、この文書の「サマリー」を参照。
  2. ヘルプ、連絡先情報、および検索の横にあるアイコン。
  3. 吹き出しのアイコン。

避ける：

  1. 読み手を誘導するアイコンや画像がない重要な説明文。
  2. ユーザーを混乱させたり圧倒したりするような、アイコンでいっぱいの雑然とした高密度のページ。

目標 2: ユーザーが必要なものを見つけられるようにする。

認知障害や学習障害があるユーザーは、必要としているコンテンツを見つけるのに苦労することがあります。また、コンテンツやタスクの中で自分の位置を確認するのに苦労することもあります。ユーザーは、探しているものをすばやく簡単に見つけることができる必要があります。ユーザーが容易にナビゲートできるように、明確でわかりやすいレイアウトにしてください。例えば：

  • 安全に関することや、ユーザーが知っておくべきことは、大量のテキストを読まなくても容易に分かるようにする！
  • ユーザーが必要なページにたどりつけるように、サイトの構造や階層を明確にする。
  • 最も重要なことをサイト内や各ページで簡単に見つけられるようにする。
  • ユーザーがページのデザインを理解しやすいように、見出し、境界線、領域を明確にした上で、優れた視覚的な手がかり（アイコンなど）を使用する。これによって、ページのナビゲーションがより容易になる。
  • ユーザーがサイト内のものを見つけやすくするために、検索機能やパンくずリストを提供する。
  • ユーザーがセクションを簡単に見つけられるように、メディアをチャンク（かたまり）に分割する。

関連するユーザーストーリー： 3.2.1 探索可能.

やるべきこと

サイト上の重要なタスクや機能を目立たせ、見つけやすくする。

これには以下が含まれます：

  • HOMEページでは、ウェブサイトの重要なタスクを目立たせる。
  • これらのタスクや機能に対して、コールアウトボックスやHOMEページのセクションを用いる。
  • 最も重要なタスクや機能を視覚的に強調する。
  • ユーザーがスクロールしなくてすむように、タスクや機能をページの上部に配置する。
  • 支援技術がすぐに見つけられるように、タスクや機能をコンテンツの最上部に配置する。
  • 主要なタスクまたは機能に対して、それぞれに有用な見出しを付ける。
  • 主要なタスクをメインナビゲーションの最上位に入れる。
メリット

遂行機能の低下、記憶力の低下、その他の認知障害や学習障害がある人は、サイト上で何ができるかを判断することが困難な場合があります。重要なタスクや機能を目立たせることによって、そのサイトが自分のニーズを満たすかどうかをより迅速に判断することができます。

例えば、チケットを購入するためにあるサイトにアクセスしたとします。レビューやその他の情報はたくさん表示されるが、チケットの購入方法が分からなければ、ユーザーはそのサイトを離れてしまいます。

詳細

重要な機能やタスクは、視覚的にもプログラム的にも目立つようにする。ウェブコンテンツ・アクセシビリティ・ガイドライン（WCAG）を参照のこと [[WCAG22]].

まずはじめに

ユーザーにとって重要なタスクは何かを考えることから始める。

例えば：

  • ユーザーが実行したい最も一般的なタスク
  • ユーザーの健康や幸福に影響を与える可能性のあるタスク

通常は、使用状況のデータから最も一般的なタスクを特定することができる。フォーカスグループや調査も、ユーザーが何を求めているかを特定するのに有効である。

事例

用いる：

  1. 最も重要なタスクは、メインページで視覚的に明確なボックスで表示する。例えば、図書館のサイトで重要なタスクとなりうるのは、検索、図書館カードの登録、支店の検索、会議室の予約などである。
  2. 最もよく使われる機能は、ページの先頭付近に配置する。

避ける：

  1. ユーザーがサイトを訪れる主な理由よりも、チームが宣伝したい項目を目立つように配置する。例えば、ある図書館のWebサイトでは、メインページには今後のイベント情報しか掲載されていない。ユーザーは、本の検索、図書館カードの登録、支店の検索、会議室の予約などを行うために、目立たないリンクをクリックしなければならない。
  2. ユーザーが必要とする可能性が高い情報を、スクロールしなければ見つけられないように配置する。

ユーザーニーズ

サイトとページの構造の両方を簡単に理解し、ナビゲートし、閲覧したい。

関連するユーザーストーリー： 3.2.3 明確なナビゲーション.

やるべきこと

サイトの階層やメニューの構造を分かりやすく、使いやすくする。

これには以下が含まれます：

  • コンテンツで取り上げるトピックを考える。次に、サイトを論理的でまとまりのあるセクションに分けて整理する。
  • メインメニューの構造に、サイトの構成を用いる。
  • サブメニューは、それが属するメインメニューの項目と明確かつ論理的に関連する項目を作成する。サブメニューの存在とそのアクセス方法が容易にわかるようにする。ユーザーが、サブメニューがどこにあるか、初めてでも正しく推測できるようにする。

以下を識別しやすくする：

  • サイトの構成
  • メニューとコンテンツの構造
  • サブメニューがあること
  • サブメニューがある場合には、サブメニューに到達する方法
メリット

ユーザーは、サイトの視覚的な階層、メニュー、構造を理解していないと、しばしば混乱し、迷子になります。明確なサブメニューと明確な構造は、サイトに何があり、どのようにそれを探せばよいかをユーザーが知るのに役立ちます。

ユーザーが戸惑うことが多いとき：

  • 構成が不明確である
  • メニューの文言が分かりづらい
  • メニューやサブメニューが見つけづらい

サイトを明確で論理的セクションに分割することが有効です。セクションを明確にし、サブセクションを見つけやすくしましょう。カテゴリ構造や見出しをわかりやすくしましょう。サイト上にあるコンテンツの要約となるようなアウトラインを作成しましょう。

メニューで使用される用語は、ユーザーが分かるものである必要があります。探しているページがどのカテゴリーに分類されるのか、ユーザーにとって明白である必要があります。ユーザーにとって馴染みのある言葉を使うことは非常に重要です。

コンテンツの階層をわかりやすくしましょう。最小限の文字サイズや文字の太さの違い、色の違いは、知覚・理解されない可能性があります。小さなデザイン要素だけに依存した階層構造は、混乱を招きます。

例えば、サブメニューのドロップダウン式アコーディオンメニューは、小さなデザイン要素によって示されているように、クリック（または「ロールオーバー」）する必要があることをユーザーが理解しなければ、表示されないかもしれません。

あるウェブページを初めて開くときは、サブメニューは折りたたまれているのが普通です。デザインによっては、サブメニューの存在を知るのが難しい場合があります。認知障害があるユーザーの中には、サブメニューがあることを推測できない人もいるかもしれません。メニューが展開するのを偶然に見たとしても、この構造がメニュー内の他の項目にも存在することを理解できないかもしれません。ユーザーがサイトの全てを利用できるように、視覚的なインジケータを追加することによって、サブメニューの項目があることに気づきやすくしましょう。

認知障害や学習障害のある人にとっては、サブメニューの項目を開くことが容易でない場合があります。例えば、メニュー項目を展開するためのコントロールは、特定のジェスチャーやその領域をマウスオーバーすることに依存しているとします。エンドユーザーは、サブメニューを展開する方法が分からなければ、タスクを放棄してしまう可能性があります。例としては、メニューテキストの特定のサイドにマウスを移動させた後にのみ展開されるメニューなどが挙げられます。

詳細

  • サブメニュー項目を示す小さなデザイン要素は、必ずしもユーザーが理解できるとは限らない。可能なかぎり、多様なユーザー層でデザインをテストして、理解しやすいものであることを確認する。

  • ユーザーが理解しやすいものにするために、一貫してベストプラクティスを用いる。

  • 小さなデザイン要素を解釈しなければならないネスト化された要素の連続は、混乱を招き、ユーザーは階層を理解できない可能性がある。

  • テキストを非表示にしている場合、または表示したり非表示にしたりできる場合は明確に示す。

まずはじめに
  1. サイト内のコンテンツの種類とカテゴリーを考える
  2. コンテンツを明確なカテゴリーに分ける
  3. 重要なカテゴリーからメインナビゲーションを構築する
  4. サブメニューがある場合は、サブメニューが閉じているときと開いているときを示す分かりやすい記号を付ける
事例

用いる：

  1. 視覚的に明確で論理的なサイトの階層構造
  2. テキストが隠されていることを、明確に示すインジケータ：
    • それを押すと追加情報が表示されることを示すために、「+」記号を一貫して使用する。
    • メニュー内で非表示になっているサブメニューの横に三角形を使用する。

避ける：

  1. ユーザーが各メニュー項目の下にある項目を正しく推測できないサイト階層構造
  2. メニューのカテゴリに対する不明確なロジック
  3. メニューの横にサブメニューの項目が視覚的に明示されていない状態：
    • マウスでサブメニュー項目の存在を発見する唯一の方法が、サブメニュー項目の上にマウスオーバーすることである。
    • サブメニューを開くにはメニュー項目の一部にマウスを置く必要があるが、この領域が不明確で、視覚的に区別できない。

ページの構造を明確にして理解可能にする［パターン］

ユーザーニーズ

サイトとページの構造の両方を簡単に理解し、ナビゲートし、閲覧したい。

関連するユーザーストーリー： 3.2.3 明確なナビゲーション

やるべきこと

ページのレイアウトを注意深く設計します。ページを理解しやすくするために、ページの構成と階層を明確にしましょう。

これには次のような方法があります：

  • ページのコンテンツを論理的なセクションに分けて整理する
  • 領域を明確に区別するために、区切り線、空白、背景色を用いる
  • 各領域の構造や目的を示すために、見出しやその他の視覚的な手がかりを提供する
  • ページ内の領域間の関係を明確にする
  • 以下を理解しやすくするために、視覚的なインジケータを用いる：
    • ページのコンテンツの構成と相対的な重要度
    • アイテムのグルーピングと関連性
    • 周囲にある情報とは異なる目的があるアイテム
メリット

認知障害や学習障害がある人は、ページの構成や関係が不明確な場合、コンテンツやアプリケーションを利用できないことがあります。ユーザーは、タスクを完了できなかったり、重要な情報を見逃したりする可能性があります。使い方が複雑で理解しにくいページには、ユーザーは戻ってこない可能性があります。

ページレイアウトが明確で整理されていれば、ユーザーは重要な情報を容易に見つけることができます。ユーザーは、ページに何が書かれているかを読み解くかわりに、自分のタスクに集中することができます。標準的なビジュアルレイアウトを使用し、要素を一貫して配置することによって、ユーザーは条件反射的に利用することができるようになります。これは、問題解決能力に影響を与える障害を持つ人、読むのが遅い人、多くのテキストを見せられると圧倒されてしまう人をサポートします。これには以下が含まれます：

良い構造： ページのコンテンツを、それぞれ明確な目的を持ったセクションに整理することによって、ユーザーは必要なセクションをより簡単に見つけ、集中できるようになります。また、ページの主目的と直接関係のないコンテンツは、明確に分離することが必要です。

ボーダーとシェーディングでグループ化する： ボーダーや明暗のコントラストなどで情報をグループ化することで、グループの識別が容易になります。

例えば：

視覚的な手がかり 書き言葉を認識・理解するのが困難な人、または集中できない人は、グラフィカルな手がかりのほうが処理しやすいと感じることがあります。一般的なグラフィカルなインジケータや視覚的な手がかりの例としては、以下のようなものがあります：

  • コンテンツの要約をグループ化する
  • 色や余白を使ったカードデザインを用いる
  • コールアウトボックスを使用するなどして、重要な情報にフラグを立てる
  • 引用部分を吹き出しに入れるなどして、異なるタイプの情報を示す
詳細

ページに多くのコンテンツがある場合、コンテンツがグループ化されていて、グループ間の関係性が分かるようになっているかどうかを確認します。

領域と明確なページ構造を作るには、以下のようなことが考えられます：

  • コンテンツのカテゴリーを明確にラベル付けし、見慣れた視覚的な手がかりやアイコンを使用する。これにより、記憶に頼るのではなく、認識と検索を助けることができる。背景色は、コントラストを強くすれば、明確な区切りとなる。
  • 見出しの構造は、文書全体の要約として機能するような、文書のアウトラインを作り出せるようにする必要がある。

アイコンは一貫して使用するとよいでしょう。また、グラフィカルなインジケータは、インターフェイスを乱雑にしないことが重要です。アイコンの数が多すぎると、ユーザーが処理する際の認知的負荷が大きくなります。明確な仕切りの例としては、コントラストの高いボーダーやホワイトスペースが挙げられます。背景色の変更も、コントラストが十分強ければ明確な区切りになります。

事例

用いる：

  1. コンテンツのセクションを適切に分離する。ホワイトスペース、ボーダー、コールアウトを使用してセクションを区切る。
  2. 見出しやアイコンを使って、コンテンツのセクションを定義する。これにより、ページ上の情報が整理され、レイアウトを理解しやすくなり、特定の情報を見つけやすくなる。
  3. コールアウトボックスを用いる。

避ける：

  1. テキストが密集していて、余白が少ない。
  2. ページ構成や階層が不明確である。
  3. 視覚的に区別されたセクションがない。
  4. セクションを定義する見出しやアイコンがないセクション。例えば、コンテンツの塊が互いにぶつかり合っている「フラットデザイン」のWebページ。認知障害のあるユーザーの多くは、どのかたまりが同じグループなのかを理解することが難しいか、不可能であると感じる。そのため、コンテンツを幾つかのかたまりに分けることの利点が失われてしまう。
  5. 論理的ではないグループやページのセクション

ページ上で最も重要なアクションや情報を見つけやすくする［パターン］

ユーザーニーズ

スクロールしたり他の操作をしたりせずに、重要な情報や必要なコントロールに到達したい。それらは隠されていたり、画面の外にあったりしない。

関連するユーザーストーリー： 3.2.1 探索可能

やるべきこと

重要なコンテンツを視覚的に目立たせます。 重要なコンテンツは、ページをスクロールしたり、コンテンツにホバーしたりしなくても、ユーザーに見えるようにする必要があります。これには以下が含まれます：

  • 重要なタスクと、それを完了するために必要なコントロール
  • 重要な機能のためのインタラクション（例：ログインフォーム、送信ボタン）
  • 重要な情報（健康への警告や安全に影響を与える情報など）
メリット

読むのが遅い人、遂行機能障害、記憶障害、その他の認知障害や学習障害のある人は、ページ上の情報や機能を見つけることができない場合があります。 例えば、たくさん読む必要があるコンテンツや、スクロールバーやマウスポインタのホバーを使う必要があるコンテンツは、見つけにくい場合があります。いろいろな画面を見て回る必要があるコンテンツを見つけることができないユーザーもいます。

ページ (または一般的なデザイン パターン) に慣れていないユーザーは、重要な情報を見つけるために目立つ視覚的なスタイリング補助に依存する場合があります。 明確な見出し構造も、読む必要があるものを減らすことでこれに役立ちます。

例えば、ある小学校では、毎週ニュースレターを発行していて、物語、活動、重要なお知らせを掲載しています。重要なお知らせの1つに、ある日学校が早く終わるというものがあります。ニュースレターには、早期下校に関する情報の前に重要ではない情報があり、重要な情報の横に警告のシンボルもありません。ゆっくり読む保護者は、重要な情報の前に読むのをやめなければならず、ある日学校が早く終わることに気づかないかもしれません。親は子どもが帰宅する時間に家にいないことになります。

別の例では、ユーザーがコメントを書いているのですが、テキストエリアにフォーカスが当たっていると送信ボタンが表示されません。その結果、ユーザーはフィードバックを送信する方法がわからなくなります。この場合、会社は送信ボタンを見つけられない人たちからは、フィードバックを受け取らないでしょう。

詳細

スクロールする前に見えるページ（コンテンツ）の量は、物理的なデバイスのサイズや解像度など、さまざまな要因に左右されます。可能であれば、サイトの統計情報を使って、ユーザーがどのような技術を使っているかを把握しましょう。このことを念頭に置いて、ページをデザインしてください。

最も制約の多いユーザーエクスペリエンス（例：幅240pxの携帯電話）を最初に考慮し、そこからターゲットを拡げて設計していきます。そうすることで、最も幅広いシナリオを考慮することができます。レスポンシブ開発の手法を採用することで、より多くのデバイスや状況に対するページの柔軟性を向上させることができます。

まずはじめに

ページ内で最も重要なものを簡単に見つけられるようにする。デザインプロセスの早い段階で、重要なコンテンツとその配置を明確にする。

ドキュメントの上部にあるスペースは、スクロールせずにユーザーに見える可能性が最も高い。重要なコンテンツはページの最上部に配置し、最大限に広範囲のユーザーにベストな体験を提供する。

事例

用いる：

  1. 重要なコントロールや機能は、ページをスクロールする必要なく見えるようにする。
  2. 送信ボタンなどの重要なコントロールは、重要度の低いリンクやボタンよりも目立たせる。
  3. 重要な情報（健康や安全に関する情報など）を強調表示して、スクロールしなくても見えるようにする。
  4. 重要な情報は、ページ上の他の重要度の低い情報から視覚的に際立たせる。

避ける：

  1. ユーザーがすぐに見つけることができない重要な情報、コントロール、または機能。例えば、ユーザーの安全に関する警告が、視覚的にはっきりせず、ページ上部に表示されていない。

メディアをチャンクに分ける［パターン］

ユーザーニーズ

欲しいものに簡単にナビゲートし、休憩を取り、ついていけなかったり気が散ったりしたときには簡単に一歩前に戻りたい。

関連するユーザーストーリー： 3.2.4 メディア

やるべきこと

論理的な構成と構造を提供し、ナビゲートしやすくする。

長いメディアは、以下のようなセグメントに分割する：

  • 論理的
  • 短い
  • ラベル付けされた
  • 特定しやすい
  • 到達しやすい、または移動しやすい
メリット

明確で論理的な構造を見出しとともに使用することで、ユーザーは注意力が散漫になったり、集中力を欠いたりしても、現在位置を見定めて、コンテンツを容易にナビゲートすることができます。これは、特に注意力に障害のある方にとって重要なことです。

短い論理的なセグメントを提供することで、ユーザーは特定のトピックを見つけ、集中することができます。ユーザーが集中力を切らしてしまった場合でも、資料の中で自分が居る場所を見つけ、覚えている最後のポイントから再び始めることができます。これは、教育コンテンツや説明書において特に重要なことです。

また、メディアをチャンク（かたまり）に分けることで、各セグメントに一意のURIを付与することができます。これにより、簡単に参照したり、共有したりすることができます。

例えば：

  • ビデオは、章やセグメントに自然に分割できることがある。
  • ポッドキャストは、1時間の録音を1つにまとめるのではなく、セグメントに分割できる。
詳細
  • 6分以内： メディアは通常、6分以下のセグメントに分割するとよい。
  • ナビゲート可能： 各メディアのセグメントへのナビゲーションを提供し、ユニークで説明的なラベルを付ける。
  • 論理的な順序： メディアのセグメントへのリンクを論理的な順序で表示する。
  • 例外： 論理的な区切りがないメディアは、細分化する必要はない。
トランスクリプトがある場合は、簡単に見つけてナビゲートできるようにする必要があることに注意する。
事例

用いる：

  1. 短い論理的なセグメントに分割されたメディア。各セクションにはラベルが貼られ、簡単にアクセスできるようになっています。例えば、30分のビデオを5つのセクションに分割し、それぞれに説明のリンクを貼って、その時点から再生できるようにする。

避ける：

  1. 分割されていないメディア。例えば、30分の動画で、分割されていないか、セクションの説明がない。ユーザーは最初から再生するか、動画内での開始位置を推測しなければならない。
  2. メディア内でラベル付けされていないセクション。
  3. メディアの概要や目次からリンクされてないセクション。

検索機能を提供する ［パターン］

ユーザーニーズ

機能やコンテンツを簡単に見つけたい。

関連するユーザーストーリー： 3.2.2 検索可能

やるべきこと

親切な検索機能を提供する。理想的には、検索機能には以下を含むべきである：

  • オートコンプリート。
  • 適切な場合には、検索結果をグループ化し、各グループに見出しを付ける。
  • 以前の検索結果を簡単に見つけることができること。
  • スペルチェック。
メリット

検索機能があれば、ユーザーはサイトのメニューが使えなくても、必要なコンテンツを見つけることができます。ユーザーは検索の使い方を学び、そのスキルを多くのサイトで再利用することができます。

メニューシステムやほとんどのサイトナビゲーションは、ユーザーにメニューのカテゴリーを理解することを求めます。遂行機能が低下しているユーザーは、正しいカテゴリーを識別できない可能性があります。

場合によっては、ユーザーは論理ではなく、記憶によって正しいカテゴリーが分かることがあります。例えば、ほとんどのユーザーは、［印刷］機能が［ファイル］メニューの下にあることをよく覚えています。記憶力が低下しているユーザーは、記憶に基づいてこれらのメニュー項目を見つけることができない可能性があります。

また、短期記憶障害や加齢に伴う物忘れのある人、注意力が散漫になりやすい人は、サイトをナビゲートしたり、コンテンツを探すために何ページも移動したりすることが困難な場合があります。あまりに時間がかかると、集中力が途切れてしまい、何を探しているのか忘れてしまうかもしれません。

検索機能は、スペルミスを修正し、適切なコンテンツや関連するコンテンツを見つけ、自動修正した検索キーワードを提案してくれるのが最も役に立ちます。

関連するトピックの検索結果が多い場合は、検索結果が適切な見出しやカテゴリの下に表示されるようにすると便利です。これにより、ユーザーは探している検索結果を見つけやすくなります。

詳細

どのページもメインページから2クリック以内で移動できるような小規模なサイトでは、検索はそれほど重要ではない。

事例

用いる：

  1. スペルチェックや提案された用語での検索が可能。

避ける：

  1. 元のキーワードとの関係によってグループ化または順序付けされていない多くの結果を提示する検索。

目標 3: 明快で理解可能なコンテンツを用いる。

一部のユーザーは言語能力に障害があります。 これらのユーザーの多くは、理解しやすい言葉を使用したコンテンツを理解しています。 たとえば、言語障害のある人は、簡単な文章や一般的な単語を理解できる場合があります。 ただし、珍しい言葉を含む複雑な言語は、彼らにはアクセスできない場合があります。

ユーザーがページのメッセージと目的を理解できるように、以下を使用します；

  • 分かりやすい単語
  • 短い文章
  • シンプルな時制
  • テキストの短いブロック
  • 明確なコンテンツ
  • 鮮明な画像
  • わかりやすい動画

優れた視覚的なレイアウトと小さなテキストのチャンクにより、コンテンツが理解しやすくなります。 分かりやすくするために、空白を使用し、前景と背景を適切に分離しましょう。 また、数値または数学的なスキルに頼ることは避けましょう。

明快な言葉を用いる［パターン］

ユーザーニーズ

語彙、構文、時制やその他のことも含めて、使われている言語を理解したい。

関連するユーザーストーリー： 3.3.1 明快な言葉

やるべきこと
  • すべてのコンテンツにおいて、一般的で明確な言葉を使用する。最も一般的な1500の単語やフレーズを見てほしい。これらは、重度の言語障害を持つ人々が知っている可能性が最も高い用語である。
  • 不要な言葉や曖昧な言葉（「など」を含む）を削除する。
  • 一般的でない頭字語、略語、専門用語は削除するか、説明する。
  • アプリケーションの中で新しい言葉を考案したり、言葉に新しい意味を与えたりしない。コンテンツを使用するために、人々が新しい言葉の意味を学ぶことを期待しないこと。新しい用語を作成する必要がある場合は、ユーザーがワンクリックなどの動作で用語説明を確認できるようにする。
メリット

これは、言語障害、処理能力障害、記憶障害のある人など、多くの人にとって有益なことです。一般的でない言葉を使うと、テキストやメディアの理解を難しくしてしまうことがあります。

言語障害のある人は、多くの場合、語彙が少ない傾向にあります。新しい用語を覚えるのは、非常に時間がかかり、困難な作業です。また、認知症の人たちのグループでは、新しい言葉を覚えることが現実的でない、あるいは不可能な場合もあります。彼らがまだ知らない一般的でない言葉を使うと、コンテンツは理解不可能（理解できない）になり、使用不可能になります。

例えば、軽度の認知症の方が、ICT冷暖房機をつけようとしたとします。暖房や冷房を選択するメニュー項目は「モード」と表示されています。この一語のために、ユーザーはユニット全体を使用することができません。このような設計が、低体温症などの緊急事態を引き起こしているのです。

一般的な言葉や用語を、最も一般的な意味合いで使うことが、こうした問題を回避することにつながります。

一般的な言葉や関連リソースについては、W3Cの開発者向けリソースページをご覧ください。

詳細

一般的でない言葉を使う場合は、次のように説明する：

  • その言葉の横に括弧で簡単な言葉を加える。
  • ポップアップで定義を表示する。
  • サポートされているマークアップを使用する（easylangを参照）。なお、easylangは新しいパーソナライゼーション仕様 [[personalization-semantics-help-1.0]] に導入される予定です。文書が完成した時には、より多くのサポートが必要です。
まずはじめに

見出し、ラベル、ナビゲーション要素、説明、エラーメッセージに明確な言葉を使う。そうすることによって、大きな時間をかけずにユーザビリティを向上させることができる。

事例

用いる：

  1. 一般的で、明確で、分かりやすい言葉や用語の定義。例えば：

    "Your landlord must follow the law."

    • 大家さんは、あなたの敷金（約束金）を、未払い家賃（あなたが支払うべき家賃）や、あなたが破損したものを修理するためなど、特定のことにのみ使用することができる。
    • 大家さんは、明確な期日までに、敷金をあなたに返還しなければならない。これは通常、あなたがアパートを出てから30日後である。
  1. 略語は、その略語が元の完全な用語よりも一般的である場合を除き、初回使用時に説明する。略語は、初回以降はtitle属性を用いたabbr要素でマークアップする。
  2. 一般的に使用されていない頭字語は、初回使用時に説明して、それ以降はタtitle属性を用いたacronym要素でマークアップする（訳注：acronym要素は廃止されているため、abbr要素を用いるべきである）。
  3. 専門用語は、避けるか、説明する。

避ける：

  1. 説明のない一般的でない言葉。例えば：

    "A Landlord’s Right to Deduct." 賃貸物件に入居する際、借主は貸主に敷金を支払います。管轄区域にもよりますが、この保証金は、借主が賃貸契約書や契約書のすべての条項とテナントに従う限り、賃貸期間の終了時に、特定の期間内に借主に返還されることになっています。以下のリンクを選択し、あなたの状況に関連する法律をお読みください。

  1. ユーザーが知らない可能性のある略語、頭字語、専門用語を説明していない。

ユーザーニーズ

語彙、構文、時制やその他のことも含めて、使われている言語を理解したい。

関連するユーザーストーリー： 3.3.1 明快な言葉

やるべきこと

最も理解しやすい時制と態を用いる。英語では通常、現在形と能動態がこれにあたります。ユーザーに直接語りかけ、動詞、文の構造の最も単純な形態を使用する。

異なる言語における最も理解しやすい時制と態を見つけるために、現地の平易な言葉（plain language）に関するガイダンスを参照する。

メリット

シンプルな時制と態を使うことは、言語障害、失読症、記憶障害のある人など、多くの人にメリットがあります。例えば、「ONボタンを押してください」（現在形、能動態）の方が、「ONボタンが押されるべきです」（受動態）よりも多くの人が理解できるはずです。

能動態では、誰が行動を起こすべきかを明確にすることができます。例えば、"It must be done." は受動態で、誰が行動しなければならないかを言っていません。"You must do it." は能動態で、誰が行動を起こすのかが明確に述べています。

また、文の目的を冒頭に置くことによって、英語の文章をより分かりやすくすることができます。現地の言語専門家は、コンテンツを理解しやすくするための言語学的なアドバイスをしてくれるかもしれません。

詳細
  • 他の態または時制の方がわかりやすい、あるいは親しみやすい場合は、そちらを使用する。
  • 現在形や能動態が存在しない、あるいは最も明確な選択肢でない言語では、最も理解しやすい時制や態を使う。
  • 過去や未来の出来事について書く場合、現在形は使わない。混乱を招いてしまうだろう
事例

用いる：

  1. シンプルな時制と言葉遣い。例えば："Your stocks went up this month."

避ける：

  1. 複雑な言葉や時制。例えば："Over the last month, we saw your stocks increasing."

多くの人は、文字通りではないコンテンツを理解できません。例えば、自閉症の人は、ジョークや比喩を理解できないかもしれません。コンテンツを親しみやすいものにするために、説明書にジョークや比喩を入れることがあります。しかし、これは、どうすればいいか分からなくなったユーザーを混乱させてしまいます。

文字通りではない言葉は、次のように説明することができます：

非テキストのメディアでは、文字通りではないコンテンツをメディアの一部として説明するか、別のファイルまたはトラックに含める。ベストプラクティスを参照のこと。

避ける：

  1. 文字通りではない文章。例えば：

やるべきこと

短いブロックの文章を用いる。

これには以下が含まれます：

  • 段落を短くする。各段落のトピックは1つだけにする。
  • 段落やかたまり（チャンク）の目的を冒頭に持ってくるようにする。
  • 短い文章を使用する。1つの文につき、ポイントは1つだけにする。
  • 箇条書きや番号付きのリストを使用する。
  • 簡潔で説明的な見出しを使用する。
メリット

テキストコンテンツをチャンク（かたまり）に分けることによって、読みやすく、理解しやすくなります。これは、処理速度や言語に関連する学習障害や認知障害がある人にも役立ちます。記憶障害がある人や、注意散漫になりがちな人にも有効です。チャンクに分けることは、マルチタスクを行っている人にも有効です。それぞれのチャンクや段落の最初に、目的や目標を置くようにしてください。

例えば、AD(H)Dの大学院生は、新しいソフトウェアのスキルを独学する必要があるかもしれません。ソフトウェアのドキュメントは、トピックごとに短い段落やリストに分けられています。その学生は、その文書が読みやすく、理解しやすいと感じるでしょう。

詳細
  • 短い段落とは？ 英語では、段落に50語以上ある場合は、2つのパラグラフに分割してみる。
  • 複数のポイントを持つ文章を書かないようにするにはどうしたらいい？　ポイントが2つ以上ある文章は、たいてい「and」や「but」などのリンクワードが2つ以上あ
  • 長い文章が、2つの短い文章よりも明確になることはあるのか？ 長い文章が短い文章2つよりも分かりやすいかどうかを再チェックすること。認知障害や学習障害がある人が長い文章を理解しやすいと思うか、ユーザビリティテストを行うとよい。
  • リストを使うとよいのは、どんなときですか？ 3つ以上の事柄を並べるときは、リストが最適です。項目、要件、例外には、（箇条書きの）順不同のリストを使用することを考えるとよい。3つ以上のステップが連続する場合は、番号付きリストにすると分かりやすい。
事例

用いる：

  1. 短いチャンクの文章。例えば：
    • カルガリーでは今週末、雪やあられがたくさん降るでしょう。運転しないようにしてください。どうしても運転しなければならない場合：
      • 冬の運転ルールに沿って、安全運転を心がけましょう。
      • 出発前に、旅行者情報のウェブサイトでどの道路が安全かを確認してください。

避ける：

  1. 長いチャンクの文章。例えば：
    • DOTDは、カルガリーに冬期天候における旅行勧告を発表。この連休中、雪や雨が降る可能性があるため、交通開発省（DOTD）は、同省職員が冬の天候に対応できるように準備していると発表しました。メンテナンス部隊は、影響を受ける橋や道路に砂や塩を撒き、道路から倒木を取り除き、必要に応じて道路を閉鎖するために待機しています。Jane Doe臨時長官は、冬の天候の脅威を真剣に受け止めるよう、ドライバーに呼びかけています。「悪天候の場合、高速道路や州間道路へのアクセスを維持するよう努めますが、雪や氷が降っている間の移動は可能な限り避けることを推奨します。」とDoeは述べています。

ユーザーニーズ

単語を音読するために必要なアクセント、文字、および付加記号を含む単語が必要である。

関連するユーザーストーリー： 3.3.1 明快な言葉

やるべきこと

文章、数字、記号などの句読点や書式は、曖昧さを減らし、読みやすさと理解力を高めるようにする。

メリット

読者によっては、単語、数字、記号の解読が自動的に行われず、ワーキングメモリや遂行機能に負荷がかかる場合があります。読者はコンテンツに負荷を感じ過ぎると、その意味を見失うリスクがあります。

ユーザーの中には、コンテンツを理解する補助として、コンテンツを読み上げる音声合成による読み上げのように、支援技術やパーソナライゼーションツールを使用する人もいます。しかし、句読点や書式によって、スクリーン・リーダーが間違って読んでしまう可能性が高くなることがあります。たとえば、ローマ数字がテキストとして読み上げられることがあります。

学習障害があるユーザーは、書式や句読点のエラーのために起こるエラーを修正するために、文字、数字、単語の間違いを変更できないことがあります。また、コンテンツを利用するためには、その意味を理解することに重点を置く必要があります。

例えば、コミュニケーションに障害があるユーザーは、音声合成による読み上げでコンテンツを聴くことができます。コンテンツが正しく読み上げられれば、理解することができます。しかし、コンテンツ、特に数字や記号が誤って読み上げられたり、読み飛ばされたりすると、理解できないことがあります。また、文字や数字、記号が見慣れないレイアウトで表示されている場合、ユーザーは混乱することがあります。

一方、コンテンツを聞いている全盲の人は、言葉が正しく発音されなくても、正しい意味を把握できる可能性が高いです。しかし、正しい意味を理解するために必要な言葉の操作（変換）は、コミュニケーションや言語に障害がある人には困難です。

詳細

言語タグを使用する 　言語タグは、曖昧さのないテキストフォーマットという目標を達成するための重要な手段です。HTML [[HTML]] の言語タグ（訳注：lang属性）とBCP 47の言語コードを参照してください。

あなたが書いている言語に応じて句読点を正しく使用する なぜなら、テキストから音声に変換されたときに、ストレスやイントネーション（プロソディと呼ばれる）のパターンが、テキストがどのように読み上げられるかに影響するためです。例えば、英語の場合、コンマやセミコロンは短い一時停止を作りますが、ハイフン - は通常無視されます。クエスチョンマーク、エクスクラメーションマーク、スピーチマークは、声の高さが上がるなど、イントネーションを変化させることがあります。

文章でローマ数字や見慣れない記号を使用することは、できる限り避けてください。　これらは読み手を混乱させ、音声合成ツールで間違って読み上げられる可能性があります。これらの記号が必要な場合は、MathMLや省略形の拡張などのテクニックを使って正しくマークアップして、追加サポートを提供するようにします。ローマ数字は、個々の文字として読まれる可能性が高いので、単独で使用する場合は大文字で表示する必要があります。

長い数字は、1桁の数字として読まれたり、1つの数字として表現される可能性があります。これは、電話番号や郵便番号で特に問題となります。これらの数字がどのように読み上げられるかを正確にコントロールすることは困難ですが、コンテンツ制作者は次のような方法で支援することができます：

  • ユーザーや支援技術がその数字の目的を認識できるようにするため、数字のコンテンツを表示して、HTMLのセマンティクスを利用します。さらに、音声合成による読み上げの改善には、次のような推奨事項があります：
  • 電話番号の場合は、その地域の電話番号に適したレイアウトを使用し、ユーザーが電話番号全体（市外局番を含む）を選択できるようにすることによって、音声合成がそのフォーマットを認識し、正しく読み上げることが可能になります。
  • 郵便番号の場合は、州や住所の情報を番号の近くに含めると、音声合成による読み上げが既知の略語（州名など）を展開でき、聞き手は文脈を認識することができます。
  • W長い数字を書くときは、どのような区切り文字が読者に馴染みがあるか、どのように読み上げられるかを考慮しましょう。一般的に英語圏では、1,000単位でカンマ、小数点の区切りにはピリオドを使用しますが、ドイツや他のヨーロッパ諸国ではその逆です。例えば、1,245は英語ではone thousand two hundred and forty-five ですが、ドイツ語では one point two four five です。音声合成の出力は、セパレータがその音声の言語のフォーマットで使用されていると仮定します。これがコンテンツと一致しない場合、聞き手は容易に混乱することがあります。1,000単位の区切り文字をスペースに置き換えることは、混乱を避けるための一般的な慣例になっていますが、長い数字をバラバラに読み上げる音声合成では、困難が生じます。例えば、120 034 943は、"one hundred and twenty, zero three four, nine hundred and forty-three" と読み上げられる可能性があります。
事例

用いる：

  1. どのような文化圏でも読んで理解できる日付。

    日付をどのように書くかを考えてください。もう一度言いますが、音声合成は音声の言語に関連した書式を使用します。04/03/2019のような日付は、米国英語の音声では「April 3rd 2019」、英国英語の音声では「4th of March 2019」と読まれます。月を単語で書き出すと、混乱を避けることができます。

避ける：

  1. 明確ではない、または文化によって読み方が異なる日付や数字。

ユーザーニーズ

単語を音読するために必要なアクセント、文字、および付加記号を含む単語が必要である。これは、アラビア語やヘブライ語などの言語の音声合成や音声リーダーに必要であることが多い。

関連するユーザーストーリー： 3.3.1 明快な言葉

やるべきこと

ユーザーが単語を正しく解読するために必要な母音、文字、または発音区分符号を含めます。これは、アラビア語やヘブライ語などの言語で必要とされることが多い。

メリット

ヘブライ語やアラビア語などの一部の言語では、オプションの母音と発音区分符号があります。これらの記号がない場合、同じ文字を持つほとんどの単語は、2つ（ヘブライ語）から7つ（アラビア語）の異なる意味を持つ発音があります。ほとんどの読者は、文脈から単語を読み、視覚的な記憶から正しい発音を推測することができます。視覚的な記憶に障害がある人、読むのが遅い人、音声合成による読み上げは、間違った言葉や発音を推測することが多いかもしれません。

例えば、言語障害のあるユーザーが、ある単語を発音しようとしているとします。このような場合、3つの異なる発音を思い浮かべてから、意味が通じる1つを見つけます。残念ながら、文脈のない言葉は、具体的な意味ではなく、アイデアを与えるのみなので、言語障害がある人の多くは意味を理解できません。音声合成では、多くの場合、これらの文字が正しい単語を話すことを求めます。

ただし、単語を正しく発音するために、すべての発音記号が必要なわけではありません。曖昧さのない発音に必要とされる文字と発音記号のみ必要となります。

詳細

単語を解読して発音することで、正しい意味を持つようになります。

まずはじめに

ヘブライ語では、正しい発音を可能にするためにYud (י)とVav (ו)を追加する。

事例

用いる：

  1. 正しい発音を可能にする追加の文字または発音記号。 例えば：
    • 彼は言う: אֹמַר /אומר (Hebrew)
    • 彼は書いた: كَتَب (Arabic)
    • 書籍: كُتُبْ (Arabic)
    • それは書かれていた: كُتِبَ (Arabic)

避ける：

  1. 必要な文字や記号がないため、ユーザーは記憶と文脈に基づいて発音を推測する必要がある単語。例えば：
    • אמר (Hebrew)
    • كتب (Arabic)

長い文書やメディアには要約を提供する［パターン］

ユーザーニーズ

長いコンテンツには、理解しやすい短い要約、または理解しやすい言葉バージョンのオプションが欲しい。

関連するユーザーストーリー： 3.3.1 明快な言葉

やるべきこと

長い文書やメディアには、簡潔な要約を提供する。

重要なキーワードがあれば強調し、文書の目的や内容を理解し、必要な情報が含まれているかどうかを判断できるようにする。

要約には、一般的な言葉、短い文章を使い、分かりやすい文体や時制で書く。

メリット

理解しやすい要約を提供することによって、そのコンテンツが自分や現在の目標に関連しているかどうかを、多くの人が素早く判断できるようになります。幾つかの文または箇条書きによるハイレベルなアウトラインが最も効果的です。要旨やエグゼクティブサマリーは、文書全体の要約を目的としているため、通常はもっと長く、もっと詳細です。

メディアの場合、要約は、注意力の低いユーザーが必要なファイルを正確に見つけ、正しいコンテンツにジャンプできるようにします。すべてのメディアファイルには、要約の説明を付ける必要があります。

詳細

中等教育レベルの読解力を持つ人が理解できるような文章の要約を提供すること。

300語以下のコンテンツでは、見出しが要約の役割を果たすことができます。

各区分の要約は、コンテンツからの主要なポイントを含むべきである。コンテンツを一意に識別して、その内容を知るために、ユーザーが要約を利用できるようにする必要があります。

段階的な説明は、言語障害、処理能力障害、記憶障害がある人など、多くの人にとって有益です。

メリット

ホワイトスペース（ネガティブスペースまたは背景色とも呼ばれる）は、乱雑さを緩和し、コンテンツに明瞭さを与えます。これにより、閲覧者はウェブページの概要を明確に把握することができます。デザイナーは、テキストやページ上のオブジェクトの位置を強調するためにこれを使用します。

ホワイトスペースを使用することで、ページ内のナビゲーションを助け、読みやすくします。ユーザーがページ上の重要な要素を見つけるのを助けることができます。認知障害や学習障害がある人にとって、ホワイトスペースは読みにくさを緩和し、コンテンツの理解度を向上させることが分かっています。

ウェブの拡張機能またはユーザー設定によって、ユーザーがオブジェクトやテキストの周りのホワイトスペースの量を調整できることも確認してください。これにより、ウェブページのコンテンツに含まれる重要な要素を識別する能力が高まります。

詳細

文字、単語、文章、段落、テキストブロックの間には明確にスペースを設ける。

ボックス、段落の見出し、コンテンツなど、オブジェクトやテキストの周りのホワイトスペースを、利用者に合わせて、かつウェブページ全体の整合性を損なわない程度に簡単に調整できるようにする。スクロールしないと重要なコンテンツが見えなくほどまでに、必要以上のホワイトスペースを追加しないようにしましょう。

なお、「ホワイトスペース」とは、背景色を意味する言葉です。常に白である必要はありません！

事例

用いる：

避ける：

  1. 密集したページ。

テキストの場合

  • テキストのブロックには無地の背景を用いる。
  • デザインが施された背景に重なるテキストには、太いアウトラインとベタ塗りを使用する。
  • 色のコントラストを強くする。

聴覚コンテンツの場合

  • 前景となる聴覚コンテンツ（会話や不要な交通騒音など）の背後に、変化の激しい背景コンテンツを置かないようにする。
  • 聴覚コンテンツの背後にある背景ノイズを除去するオプションを提供する。
メリット

文章をフレーズ単位で読むと、単語単位で読むよりも意味が伝わりやすい。また、フレーズは理解しやすい。一度に処理できる単語が多ければ多いほど、読むスピードが上がり、書かれていることを理解しやすくなり、興味を持続させることができます。多くの人は一度に一行以上の文章を理解することができます。多くの人にとって、一度に多くの単語を読み取ることは、理解するために必要なことなのです。読むのが遅い人は、1つの文章を6～9回の視線移動でゆっくり読み、時には1つの単語（または単語の一部）だけを一度に取り込むことがあります。背景を追加すると、読者が読み取れる単語の数が減ります。背景を消すと、より多くの単語を同時に理解することができます。

また、読解には単語の自動認識もよく使われます。例えば、英語には従来の文字音パターンに当てはまらない単語が約200個存在します。これらの単語を記憶し、自動認識する必要がある。これらの単語をユーザーが認識できなければ、テキストを理解することは難しくなります。背景があると、ユーザーが単語を認識するのにかかる時間が長くなります。

同様に、音声トラックの背景ノイズは、メインコンテンツを処理して理解することを難しくします。

ユーザーニーズ

テキストの意味を理解したい。ジョークやメタファーを誤解する可能性があるので、説明されていない、暗示されている、または曖昧な情報は欲しくない。

関連するユーザーストーリー： 3.3.2 視覚的な提示.

やるべきこと

次のような暗示的な情報や曖昧な情報に対して、定義や説明を提供する：

  • 身振り手振り
  • 感情
  • ジョーク
  • 皮肉
  • 隠喩と比喩
  • 顔の表情

これらの定義や説明は、暗示的なコンテンツの近くにあるテキストか、マークアップで提供されるべきです（ベストプラクティスを参照）。

メリット

暗示的な内容は、意味が明確でないため、一部のユーザーにとって困難な場合があります。これには、抽象的な内容、皮肉、または比喩が含まれます。意味が明確でないため、ユーザーに理解するための知識を要求します。

身振り手振り、感情表現や顔の表情だけで何かを伝える場合、すべてのユーザーが理解できるように、別の方法でこれを含めることが重要です。その一つの方法として、補足テキストがあります。

例えば、ソーシャルメディアの投稿で、ある人の本音を伝えるために画像を使用するとします。しかし、その画像が示す感情を理解できない人がいるかもしれません。より多くの文脈がなければ、作者が伝えようとしていることを見逃してしまうのです。

同様に、ある研究では、自閉症の人々に、暗示的な内容が多く含まれる映画を見てもらいました。彼らは俳優の口元を見ていたが、皮肉などの情報は彼らの表情によって伝えられていた。映画の中で何が起こったかを尋ねると、暗示的なコミュニケーションと台詞のポイントを見逃していた人がいました。

詳細

これには以下が含まれます：

  • 何かが重要であること、または記憶されるべきことを示すために単独で使用されるグラフィック。
  • テキスト中の皮肉
  • 重要性を付加したり、対になるテキストの文字通りの意味とは異なることを伝えるために使用されるアニメーション。

なお、標準的な絵文字には、説明や代替テキストが付属していることが多い。

事例

用いる：

  1. 読者がコミュニケーションの意図を理解できるように、メールやソーシャルメディアへの投稿で皮肉を書く際に付与する「（皮肉）」などの補足テキスト。
  2. （将来的に）文字通りではないテキストの代替を追加するパーソナライゼーション・セマンティクス。[[personalization-semantics-help-1.0]] を参照のこと。

避ける：

  1. 説明されていない比喩や類語。例えば、「あなたは私を幸せにしてくれる」ではなく、「あなたは私の太陽です」。

やるべきこと

数字や数値の概念に代わるものを提供する。

メリット

すべての人が数字や数値の概念を理解できるわけではありません。

例えば、数学に特化した学習障害である「ディスカリキュア」の人もいます。ディスカリキュアの人は、数字や数学的な概念に大きな問題を抱えていますが、他の知的分野では優れていることが多いのです。

例えば、ディスカリキュアのユーザーは、温度データが数字のみで表示されると処理に困ることがあります。しかし、数字以外の選択肢（冷たい、暖かい、熱いなど）が用意されていれば、内容を理解することができます。

計算能力の問題はさまざまな障害が原因で発生する可能性があり、最も深刻なのは数字を読んだり理解できないことです。また、相対的な大きさや時間など、あらゆる計算が苦手な人もいます。

例えば、90cmという長さの概念は理解できても、0.9mと900mmが同じ長さであることに対応するのは難しい場合があります。

他の例としては、電車の時刻表には、毎時00分に発車する電車の相対的な時刻が記載されています。ユーザーは、自分の場所から次の電車がいつ発車するかを計算することができません。

詳細

数学の理解がコンテンツを利用する主な要件でない場合は、以下のいずれかを使用してください：

  • 数字を非数値的な概念で補強する。例えば、とても寒い、冷たい、涼しい、穏やかな、暖かい、熱い、とても熱い。
  • 将来的には、パーソナライゼーション・セマンティクスを使用して、非数値的な概念を追加することも可能です。 [[personalization-semantics-help-1.0]] を参照のこと。

他のユーザーは、長いテキストよりも数学の方が理解しやすいと感じるかもしれないことに注意してください。

コンテンツに数学のスキルが不可欠な場合：

  • 拡張可能なデジタル数学に移行する（画像内の数字ではない）。
  • 議論されているセクションのハイライト表示を有効にする。
  • 数字の部分と一緒に読むことができる特別なヘルプにリンクする。
  • 数学のセクションを、ユーザーの好みに応じて単語や要約に置き換えることができるようにする。

    • パーセンテージ

ユーザーがミスを回避できるようにする必要があり、もしミスが発生したら簡単に修正できるようにする必要があります。

多くのユーザー、特に認知障害や学習障害がある人にとって、フォームに入力することは困難です。良いデザインであれば、ミスは起こりにくくなります。

関連するユーザーストーリー： 3.1.2 明確な操作

やるべきこと

ユーザーがを始動しない限り、コントロールとコンテンツは所定の位置に留まり、動かないようにします。

ユーザーがアクションを起こしたり、ウィンドウサイズなどのデバイスのプロパティを変更したりすることによって、動き始めることがあります。.

これは通常、以下の方法で実現できます：

  • ページの読み込みや更新の際に、コントロールやコンテンツが動かないようにする。
  • ページの読み込み中にコンテンツが動いたり変更されたりした場合は、「読み込み中」のインジケータを明確に表示する。
  • ユーザーが操作しない限り、リスト内のアイテムの位置など、コンテンツを更新したり動かしたりしない。
メリット

コントロールが動くと、手と目の協調が遅いユーザーや認知処理速度が低下しているユーザーは、間違ったコントロールを押す可能性があります。 これにより、不要なアクションやエラーが発生します。 ユーザーは、方向感覚の喪失、混乱、さらにはコンテンツの誤った理解を経験する可能性があります。

例えば、あるユーザーが動画のボタンを押そうと動いたとします。その際、誤ってデバイスをなでたとします。すると、画面が横向き（ランドスケープ）になって、コントロールが動いてしまう。ユーザーはアイトラッキングや手と目の連動が遅いため、新しいビデオへのリンクを押してしまう。

また、コントロールやコンテンツを移動することによって、認知的な過負荷を引き起こし、精神的な疲労を増加させる可能性もあります。例えば、外傷性の脳損傷があるユーザーがコンテンツを読んでいると、コンテンツが更新され、現在のコンテンツの上に追加の記事が表示されます。このとき、ユーザーが読んでいる記事は下に移動します。このような場合、ユーザーは混乱し、アプリケーションは非常に使いづらく、理解しづらくなります。

詳細

予期せず動くコントロールには、次のようなものがあります：

  • 位置を変えるリスト内のリンク
  • 画面の向きの変化
  • ユーザーが完了したと思っている、ページの遅い読み込み
事例

用いる：

  1. ページが読み込まれたり更新されたりしても、移動しないコントロールやコンテンツ。
  2. ページが読み込まれている間はっきりと見える、読み込み中であることを示すアイコン。コンテンツの読み込みが終了し、それ以上動きがない場合は、アイコンを削除する。
  3. ユーザーが変更を開始したときにのみ、動くコントロールとコンテンツ。

避ける：

  1. ユーザーの要求がなくても動くコントロールやコンテンツ。例えば、ユーザーが電話をかける電話番号を選ぼうとしているとする。ユーザーが電話番号に触れようとすると、電話番号が下にずれる。ユーザーは間違った電話番号を押してしまい、間違った相手に電話をかけてしまう。
  2. 位置が変わるリスト内のリンク
  3. 簡単に無効にできない、画面の向きの変更。
  4. ユーザーが完了したと思っている、ページの遅い読み込み。

ユーザーニーズ

ミスをする前に、その前はどこにいたのかを正確に知るために、予測可能な戻る機能または元に戻す機能が欲しい。

関連するユーザーストーリー： 3.4.2　取り消す（Undo）

やるべきこと

常にユーザーが前のポイントに戻れるようにする。

標準の［戻る］ボタンは、ユーザーにとって馴染み深いものなので、これを行うのに最も適した方法です。多くのユーザーは、まず［戻る］ボタンを試してみるでしょう。

ユーザーが［戻る］を押しても、決して作業中の内容を消去してはいけません。

メリット

ユーザーが前のポイントに戻れるようにすることによって、ミスを防ぎ、ミスが発生したときにも簡単に修正できます。

ミスの例としては、以下のようなものがあります：

  • 誤ってコントロールに触れてしまう。
  • 誤って新しいリンクを開いてしまう。
  • 開いたままにしておこうと思ったウィンドウを閉じてしまう。

ミスをしやすい人、ミスが多い人は、自分の作業や以前の選択が削除されることなく、戻って変更できることが重要です。

例えば、あるユーザーがビデオを見ているとします。音量を上げようとしたところ、別のリンクに触れてしまいました。すると、新しいビデオが読み込まれました。ユーザーは、［戻る］ボタンを押して、以前見ていたビデオに戻ることができました。ユーザーは、音量を上げようとしたときに、もしミスをしても、簡単に戻ってやり直せることが分かりました。

別の例では、［戻る］ボタンが期待通りに動作せず、どこか別の場所（HOMEページなど）に移動してしまったとします。音量を変えたり、コメントを入れたりしようとすると、見ていた動画が消えてしまい、元に戻す方法が分からなくなることがよくあります。ユーザーは、再びメインコンテンツが消えてしまったらと思うと、ウェブサイトのどの機能も使えないと感じるようになった。そうなると、画面を拡大したり、音量を変えたり、コメントを残したりしなくなります。

フォームの場合、ユーザーがデータを再入力しなければならない時は、毎回新たなミスが発生する可能性があります。認知障害や学習障害がある人にとって、データの入力や再入力はストレスや疲れを感じる場合があります。そのため、ミスが発生する可能性が高くなり、正しいデータを送信して目的のタスクを完了することが不可能になるかもしれません。

不安や記憶困難、指示に従うことに困難がある人にとって、入力した情報をさかのぼって確認できる機能は非常に重要です。例えば、指示に従うことと、自分の答えを確認することを、別々の作業として行うのが最適な人もいます。指示に従うことに集中して入力することが出来て、後で答えを確認するために振り返ることができれば、より効果的です。

まずはじめに

間違って送信した場合でも、ユーザーが入力したデータを戻って確認する機会を提供する。［戻る］ボタンは、常に期待通りに機能する。

詳細

ユーザーが戻ることをサポートするオプションには、次のようなものがあります：

  • 明確にラベル付けされたアクションによって、ユーザージャーニー内の前のステップに戻る。
  • クリック可能なパンくずリストを使用し、前のステップをクリックしてもデータが失われないようにする。
  • データを失うことがない［戻る］や［元に戻す］機能を用いる。
  • 将来的には、パーソナライゼーション・セマンティクスを使用して、ステップのログを残して、プロセス内にあるステップに戻ることも可能になる。 [[personalization-semantics-1.0]] を参照のこと。
  • 閉じたウィンドウやオプションを再び開く。
事例

用いる：

  1. 前に戻りやすくするデザイン。例えば：
    • ユーザーが動画を見ている。誤ってコントロール（ボタン類）に触れてしまった。［戻る］を押すと、さっきまで見ていた再生位置に戻ることができる。
    • あるユーザーが、求人に応募するためにオンラインフォームに入力している。誤って［HOME］アイコンを押してしまい、フォームから離れた場所に移動してしまった。このとき、［戻る ］ボタンを押すと、データを失うことなく元の位置に戻ることができる。
    • ユーザーは、誤解しているセクションがないか、回答し忘れていないかを確認するために、すべての画面に戻ることもできる。また、入力ミスをしたデータを編集することも可能である。

避ける：

  1. 前に戻りづらいデザイン。例えば：
    • ユーザーが動画を見ている。誤ってコントロールに触れてしまい、新しいビデオに移動してしまった。［戻る］を押すと、新しい動画が小さくなるが、元の動画には戻してくれない。
    • 就職活動中にオンラインフォームに入力する。ユーザーは、質問に答え忘れていないか確認するため、前の画面に戻る。このとき、［戻る］ボタンを押すと、以前に入力したデータはすべて消去されてしまう。

タスク開始時にユーザーに料金を知らせる［パターン］

ユーザーニーズ

作業を始める前に、必要な情報（クレジットカード、住所など）を知らせてくれるなど、タスクを管理するためのサポートが欲しい。

関連するユーザーストーリー： 3.4.1 アシストとサポート

やるべきこと

取引開始時に、典型的な価格を含むすべての料金についてユーザーに伝える。あらゆる条件や条項も、取引開始時に簡単な言葉で提供すべきである

メリット

記憶力、細部への注意力、読解力に問題のある認知障害や学習障害があるユーザーは、取引タスクの開始時に明示されない限り、料金に気づかない可能性があります。利用規約はリンク先でも構いませんが、料金は明確に表示し、理解しやすい言葉で提供する必要があります。

販売開始時に料金を明確にすることは、すべてのユーザーに利益をもたらします。特に、認知障害や学習障害のあるユーザーは、料金が含まれていることを推測したり、推定したりする可能性が低いため、恩恵を受けることになります。このような人々は、ホームページや指定された料金ページなど、ユーザーフロー内にある他の場所を見ることを知らないかもしれません。

遂行機能や記憶力が低下している人が、十分な情報を得た上で意思決定するためには、すべての結果が整然と提示されている必要があります。料金が明確でない場合、取引の同意が不明確になります。

また、障害があるユーザーが購入の手続きをする際には、非常に時間がかかることがあります。オンラインで商品を購入しようとして時間を費やした後に、購入できないことを納得するのは、はるかに困難であり動揺してしまいます。値段を理解していなかったと自分を責めることも多く、自信を喪失してしまうかもしれません。そして、ネット上での日常生活において、自分自身を信頼することができなくなってしまうかもしれません。

例えば、遂行機能に問題がある人は、航空券を注文しようとして、税金、国際線料金、手荷物料金など、元の料金に含まれていない追加料金があることに気付かないことがあります。何時間もかけて休暇を予約しようとした後で、予算に余裕がないことに気づくかもしれません。あるいは、予算を超えたものを買ってしまうこともあります。また、過去にこのプロセスを経験したことがあっても、その経験をもとに最終的な価格を予想できないこともあります。その結果、ユーザーはオンラインで旅行商品を自力で購入することに自信をなくしてしまう。その結果、ユーザーは支払うことのできない負債を抱え、二度と購入しようとしなくなるか、もしくは（旅行代理店やアシスタントなどの）専門家の助けを借りなければ購入できなくなります。

事例

用いる：

  1. 料金の明確な表示。ユーザーはすべての料金を知ることができ、商品の購入を決定し、ショッピングカートに入れる際に、十分な情報を得た上で決定することができる。
  2. 可能性がある料金の範囲の明確な記載。送料が変動する商品については、送料の範囲と、送料を変動させる要因を記載し、さらに詳細を確認できるリンクを貼る。例えば、「重量や配送スピードが配送料に影響し、地域によって4ドルから400ドルかかる場合があります。」
  3. 抜き打ちの条件はない。

避ける：

  1. 新しい料金や隠れた手数料が含まれ、結果として予想以上の合計金額を請求される最終取引。
  2. タスクの最初からユーザーにとって明確でない購入条件を含む最終取引。
  3. ユーザーが購入に多くの労力を費やすまで知らなかった料金や条件が含まれている取引。
  4. 完了時に合計金額でユーザーを驚かせる取引。

ミスを防ぐフォームを設計する［パターン］

ユーザーニーズ

失敗を回避できるインターフェイスにしてほしい。

関連するユーザーストーリー： 3.4.1 アシストとサポート

やるべきこと

ユーザーがミスをする可能性を減らすようなフォームデザインを選択する。これには以下が含まれます：

  • できるだけ少ない情報を入力させる。
  • 必須項目を明確にする。
  • テキストフィールドでは、できるだけ多くの形式を受け入れる。例えば、電話番号の場合、さまざまな形式を受け付ける。
  • 長い番号は分割して入力する（フィールドをまたがるオートコンプリートに対応する）。
  • 有効な入力だけが選択されるようなインターフェイスを使用する。
  • オートコンプリートやフォームコントロールのパーソナライズを使用する。
  • オペレーティングシステムでサポートされている場合は、音声プロンプトを受け入れる。
  • 可能かつ信頼できる場合には、入力エラーを自動的に修正する。
  • ユーザーに既知の提案や修正方法を提供する。
メリット

認知障害や学習障害、加齢による物忘れのあるユーザーは、何度もエラーを起こしているうちに、タスクを放棄してしまい、完了できないと思い込んでしまうことがよくあります。エラーメッセージが分かりにくいこともあります。エラーの訂正は、ユーザーにとっては困難でフラストレーションのたまるものであり、認知的疲労を増大させることが多い。多くのユーザーは、連続したエラーに遭遇すると、中断せざるをえません。

例えば、オンラインバンキングに登録する際、ユーザーの生年月日を入力するフォームがあります。必須の入力形式は "xx/xx/xxxx" で、一桁の場合は先頭にゼロを付けることを求めています。もし、入力修正のない単一の入力フィールドが表示された場合、認知障害のあるユーザーは "1/3/1996" と入力し、エラー通知を受ける可能性があります。（入力フィールドの下やエラー通知に入力形式が表示されていても）求められている入力形式が "01/03/1996" であることが、ユーザーにとって明確でない場合があります。

よく設計されたフォームは、情報の記入を容易にし、正しい日付形式を自動的に修正または提案することによって、ユーザーのミスを防止します。

自動修正によってユーザーが発生させるエラーを最小限に抑えることは、エラー通知を最小限に抑えることにもつながります。エラー通知は、疲れるし、気が散って、タスクやタスクの完了から集中力を奪ってしまうかもしれません。

詳細
  • 必須の内容を明確に示す。
  • 修正内容が確実な場合のみ、エラーを修正する。そうでない場合は、修正候補がわかっているのであれば、その候補をユーザーに伝える。
    • 例えば、「2月1日（01/02）と1月2日（02/01）のどちらでしょうか？」
  • カレンダーと日付
    • カレンダーは、関連する最初の日にデフォルト設定する必要があります。 勤務カレンダーは、ユーザーの国・地域における最初の営業日の曜日にデフォルト設定する必要があります。
    • カレンダーベースの予約システムは、ユーザーが出発日よりも前に帰国日を予約することを防がなければなりません。
  • 気温
    • ユーザーの所在地のデフォルトの温度フォーマットを使用する。
事例

用いる：

ミスを少なくするデザイン。例えば：

  • 市または州の情報とともにテキスト フィールドに書き込まれている郵便番号のエラーを修正する。
  • ユーザーが不適切な日付が選択しないようにして、選択しようとした場合は分かりやすく説明する。

避ける：

ミスしやすいデザイン。例えば：

  • 予約フォームに2つのカレンダーがあり、明確なラベルや説明がない。ユーザーは、6月1日出発、5月30日帰着など、可能かどうかわからない日付を選択できる。
  • システム上、ユーザーは警告なしに不適切な日付を選択することができる。カレンダーは、不適切な日付がグレーで表示されるだけなので、気づかない可能性がある。警告が表示されない。

Mフォームのエラーを簡単に元に戻せるようにする ［パターン］

ユーザーニーズ

自分の作業をチェックして、今やった作業を失うことなく前に戻りたい。

関連するユーザーストーリー： 3.4.2 取り消す（Undo）

やるべきこと

常にユーザーが自分の作業を確認し、ミスを修正できるようにする。ユーザーがミスを修正した後は、追加のステップをやり直すことなく、元の場所に簡単に戻ることができるようにする。

金銭的な取引や重要な情報については、ユーザーが簡単に取引をキャンセルできるようにする。ユーザーが取引をキャンセルできる時間など、重要な情報については、明確な情報と簡潔な説明を提供すること。

メリット

認知障害や学習障害がある人は、フォームの入力時に他の人よりも多くのミスをします。ミスを簡単に修正できない場合、彼らはタスクを完了することができません。

ミスを元に戻す機能があれば、認知障害や学習障害がある人はフォームを安全に使用することができ、ミスから生じる影響を減らすことができます。

例えば、記憶障害があるユーザーは、ショッピングカートに商品を追加したことを思い出せず、同じ商品を2回追加することがあります。また、旅行の予約の際に日付を間違えたり、その他のミスを犯すかもしれません。

認知障害や学習障害がある人にとって、自分の仕事をチェックし、ミスを簡単に修正できる機会があることは不可欠なことです。

認知障害や学習障害がある人にとって、ミスが理論的に取り消せるだけでは十分ではありません。多くの場合、トランザクションを取り消すプロセスは複雑で、手助けなしでは対処できません。しかし、そのような手助けを受けることができない場合、自分が犯したすべての間違いを背負って生きていかなければならなくなります。さらに、ミスを修正するプロセスが難しすぎると、ユーザーは取引を中止したり、不要な商品を購入したりする可能性があります。

このようなことが何度も繰り返されると、その影響は計り知れない。その結果、障害のあるユーザーの多くは、多くの作業でインターネットを使わなくなる可能性があります。

ユーザーがいつでもショッピングカートの中の商品の数を変更できるようにすると、ミスを大幅に減らすことができます。

最終的な送信の前に、商品の数量やその他の費用を含む注文の要約を表示することで、ユーザーはミスを発見して、注文を変更する機会を得ることができます。この例では、購入内容の要約によって、ユーザーは数量にミスがあったことと、合計金額が予想以上に高かったことを確認することができます。

場合によっては、ユーザーは最終的にデータを送信した後にミスに気づくこともあります。取引をキャンセルする方法を簡単な言葉で説明し、ユーザーが取引をキャンセルするために必要とされる時間を理解できるようにします。そうすることによって、詐欺に遭いにくくなります。

例えば、注意欠陥・多動性障害を持つユーザーがWebサイトで旅行券を購入する場合、詳細を理解するのに苦労し、注意力が低下している可能性があります。注文が正常に完了するかどうかは、プロセス中の複数のステップで提供される情報に依存しています。請求先住所の番地や郵便番号にミスがあると、注文は成立しません。最終的な注文を送信する前に要約が提供されない場合、ユーザーは支払いが拒否された理由を理解できず、注文をあきらめる可能性があります。また、訂正できる方法が不明確な場合、ユーザーは注文をやめてしまう可能性があります。

詳細

これは通常、以下を含みます：

  • 変更：　ユーザーは、すべてのデータを簡単に確認して、自動的に識別されない可能性のある間違いを含め、ミスを修正できるのはシンプルなことです。ユーザーは、明確にラベル付けされたアクションによって情報を変更し、データを不要に失うことなく、1つの明確にラベル付けされたアクションで、元の場所に戻ることができます。（変更した内容に関係がある場合は、一部のデータを入力する必要があるかもしれません）。
  • 確認済み：　重要な情報を送信する前に要約を説明し、最終的な情報を送信しようとするときにユーザーに伝える。
  • 取引中止の期間や指示も明確で分かりやすい。
まずはじめに

ミスが経済的損失や脆弱性など重大な結果をもたらす可能性があるフォームから始める。

事例

用いる：

  • 重要な情報を送信する前に提供される要約。ユーザーは、情報を修正し、ワンクリックで要約に戻ることができます。
  • ユーザーが前のステップを見たり、戻ったり、変更したりすることができるクリック可能なパンくずリスト。

避ける：

  • 重要な情報を送信する前に要約が提供されるが、ユーザーが訂正すると、入力した他のデータが失われてしまう。

明確なラベルを画面に表示する［パターン］

ユーザーニーズ

明確なラベル、ステップバイステップの指示、明確なエラーメッセージにしてほしい。

関連するユーザーストーリー： 3.4.1 アシストとサポート.

やるべきこと

明確なラベルを使用する。ラベルは：

  • 一般的な言葉や理解しやすい言葉を使用すること。
  • 関連するコントロールの横に表示すること。
  • 認知障害や学習障害がある人が使用するものを含む支援技術で読み取り可能であること。
メリット

ラベルがない、あるいは不明確な場合、ユーザーはその機能が利用できること、あるいはそのコントロールが何であるかが分からないことがよくあります。多くのユーザーはコントロールが何であるか推測できますが、記憶力や遂行機能に問題がある認知障害や学習障害のあるユーザーは、デザインパターンを覚えていない可能性が高く、コントロールが何であるかを理解することができません。馴染みのある用語を使用した、コントロールの横にある明確なラベルは、認知障害や学習障害がある人の助けになります。

同様に、ラベルがコントロールの隣にない場合、一部のユーザーにとって混乱を招くことになります。コントロールの横にラベルを置くことができない場合は、ラベルとコントロールを視覚的に明確に関連付けるインジケータを用意する必要があります。これには、認知障害や学習障害を持つユーザーによるユーザーテストが必要で、使い勝手が良いことを確認する必要があります。

例えば、初期段階の認知症を患っているユーザーがアプリケーションを使用しているとします。いくつかのコントロールには視覚的なラベルがありません。介護者がそのコントロールが何のためのものかを示せば、アプリケーションを使用することができます。翌日、彼らは再びそれを使おうとしますが、コントロールが何のためのものであるかを覚えていません。彼らは、このアプリケーションを使えないことになります。

別の例では、フォーカスを外すとラベルが消えます。ユーザーはそのコントロールが何であるかを思い出せず、再表示させる方法も知りません。

ラベルは、表示されていて、かつ支援技術で読み取ることができ、ラベル付けされたコンテンツの近くにある必要があります。

詳細

認知障害や学習障害がある人の多くは、ウェブの拡張機能やシンプルな音声合成を利用しています。これらの支援技術は、しばしばWAI-ARIA [[wai-aria-1.2]] または title属性を読み上げません。それが変わるか、拡張機能がそれらを表示するまで、ラベルは、認知・学習障害者のためにこれらの属性に依存してはなりません。

事例

用いる：

  1. 簡単な一般的な単語を使用し、コントロールの横にあり目に見えるラベル。例えば：
    • first name ____________________

避ける：

  1. 非表示のラベルや一般的でない言葉を使ったラベルで、理解しにくい。どのようなアクションが必要なのかが不明確である。例えば：
    • given name ___________________

明快で段階的な説明文にする ［パターン］

ユーザーニーズ

明確なラベル、ステップバイステップの指示、明確なエラーメッセージが必要で、そうすれば何をすべきかがはっきりとわかる。

関連するユーザーストーリー： 3.4.1 アシストとサポート

やるべきこと

次のように明確な説明文を書くこと：

  • フォームのフィールドやアクティビティの前、または隣にある。
  • ステップごとに分かれていること（ステップが省略されていないこと）
  • 明確で、簡潔で、アクセシブルである。
  • 何をすべきか理解しやすいように、例やイラストを用いる。
メリット

明確な説明文は、ユーザーのミスを防ぐのに役立ちます。これにより、フラストレーションが軽減され、助けを求めずに済むため、ユーザーの自律と自立性が高まります。これは、認知障害や学習障害がある多くの人々だけでなく、異なる文化圏の人々、新興市場、Webフォームに慣れていない、または文化的背景を見逃す可能性のある新しいユーザーにも役立ちます。

例えば、加齢による物忘れがひどい人がフォームに入力しようとしているとします。住所と郵便番号を1行にまとめて入力します（手紙を書くときと同じです）。すると、エラーメッセージが表示されます。何度もエラーメッセージが表示されると、疲れ果ててしまい、フォーム入力を完了できなくなります。

詳細

エラーメッセージだけでなく、プロセスの開始時に説明文を提供すること。

ユーザーがタスクを完了するために必要な説明文を提供すること。複数のフォーマットを受け入れる場合や、エラーが自動的に修正される場合には、ユーザーがタスクを完了するために必要となる説明文は少なくなります。

なお、説明文は見慣れたアイコンの後ろに隠されていることがあります。

まずはじめに

エラーがよく起こるシステムでは、最も影響力のあるエラーにまず取り組み、必要に応じてガイダンスを追加していく。

事例

用いる：

  1. 明確で分かりやすい説明文。例えば：
    • ユーザーが入力すべき番号を示すために、番号が強調表示されたパスポートの画像を提供すること。
    • カレンダーコントロールにおいて、どの曜日が週の初めであるかを明示する（例：日曜日または月曜日）。

避ける：

  1. 複雑な作業に対する明確な説明文がない。例えば：
    • パスポート番号を要求するが、パスポートに記載されている複数の番号のうち、どれが必要なのかが示されていない。
    • あるサイトが週の初めの曜日を明確にせず、週は月曜日から始まると仮定している。異なる文化圏のユーザーが、週は日曜日から始まると思い込んでいて、ミスをする。

様々な入力書式を許容する［パターン］

ユーザーニーズ

異なる書式を受け入れ、それらを間違いとしてマークしない入力フィールドにしてほしい。

関連するユーザーストーリー： 3.4.1 アシストとサポート

やるべきこと

通貨、タイムゾーン、ロケール、住所、クレジットカード番号などの値のテキスト入力では、すべての書式のバリエーションを受け入れる。

メリット

フォーム入力のプロセスを寛容にすることによって、ユーザーは、大量のエラーを発生させることなく、入力できるようになります。また、エラーで困ったときに助けを求めずに済みます。これにより、ユーザーの自律と自立性を高めながら、フラストレーションを軽減することができます。

これは、学習障害や認知障害、加齢に伴う物忘れがある人にもメリットがあります。また、異なる書式に慣れている方にも役立ちます。

例えば、加齢による物忘れがあるユーザーが電話番号を入力する際、ハイフンを入れて入力したとします。すると、システムがその形式を受け付けないため、エラーメッセージが表示されます。電話番号を忘れたのか、それとも別の間違いをしたのか、疑問に思うでしょう。そして、そのフォームを使うのをやめてしまうのです。

事例

用いる：

  1. 柔軟な入力タイプおよび検証のためのプラットフォーム。
  2. 国際的なバリエーションを含む、入力される可能性のある様々な書式のリサーチとサポート。
  3. 最小限の必須項目。
  4. 異なる書式を受け入れる柔軟な入力フィールド。例えば、次のようなものを受け入れる入力フィールド：
    • 様々な通貨記号
    • クレジットカード番号（スペースあり・なし）
    • 多くの地域で使用されている電話番号。国番号、地域番号、およびオプションの括弧を用いた番号。
    • アクセント記号付きの文字など、国際的な文字。

避ける：

  1. 入力を任意の長さに制限する。
  2. 特定の区切り文字が必要でなく、無視できる場合でも、その区切り文字に固執する。
  3. ユーザーが使用する可能性のある書式を受け付けない入力フィールド。例えば：
    • ユーザーが使用する通貨ではない可能性がある特定の通貨値を強制的に使用させる。
    • カードにはスペースを含む番号が印刷されているにもかかわらず、スペースを要求しないクレジットカード番号のフィールド。
    • 電話番号のフィールドは、"+"コードや括弧を受け付けない。

データ消失と「タイムアウト」を回避する［パターン］

ユーザーニーズ

自分の作業を完了するための時間が必要。郵便番号や社会保障番号など、必要な情報を探すためにセッションがタイムアウトするのは避けたい。

関連するユーザーストーリー： 3.4.1 アシストとサポート

やるべきこと

タイムアウトを避け、ユーザーが作業中に作業内容を保存できるようにする。

これが不可能な場合は、ユーザーがプロセスを開始するときに知らせる：

  • プロセスを完了するために利用可能な時間。
  • タイムアウトが発生した場合、入力したデータが失われる可能性があること。
メリット

時間制限のあるイベントは、認知障害や学習障害があるユーザーにとって大きな障壁となる可能性があります。これらのユーザーは、コンテンツを読んだり、オンラインフォームに入力するなどの機能を実行するために、より多くの時間を必要とする場合があります。ヘルプを読んだり、メモを見たりする必要があるかもしれません。

認知障害や学習障害のあるユーザーは、取引を完了するために必要な情報を調べるために、さらに時間が必要な場合があります。また、プロセスの中で自分の現在位置を見失わずに、すでに入力されたデータを失うこともなく、休憩する必要がある場合もあります。

例えば、eコマースサイトで買い物をするとき、ユーザーは必要な情報を覚えていないことがあります。日付、電話番号、郵便番号など、認知障害や学習障害がないユーザーにとっては覚えやすいものかもしれません。このような場合、ユーザーは情報を調べる必要があり、画面から離れている時間がかかります。そして、その情報を慎重にフォームにコピーする必要があります。

また別の例では、ユーザーが、ホテルの予約や航空券の購入などのために、オンライン手続き完了させようとしているとします。この場合、ユーザーは説明文およびプロセスを完了するために必要なデータ入力の量に圧倒されます。一度に完了することができないため、一旦休憩を取ります。

疲れてくると、ユーザーの認知能力は低下します。すると、その日のタスクを中断しなければならなくなります。データが消えないと分かれば、精神的な疲労も回復させてから、タスクを成功させるために戻ってくることができます。

ここで重要なのは、多くの人が「タイムアウト」の通知を読むのに時間がかかるということです。ユーザーが時間延長の方法を読み終える前に、セッションが終了してしまうことがよくあります。ユーザーが情報を調べている場合は、タイムアウトの通知に目を向けないでしょう。

詳細

機密情報（クレジットカード情報など）を入力または表示させるため、Web サイトがタイムアウトしなければならない場合、Web サイトは最後の段階で機密情報の入力を要求する必要があります。

また、クレジットカード情報を入力したら、セッションがタイムアウトする可能性があるため、速やかに処理を完了するよう、Webサイトはユーザーに警告する必要があります。

事例

用いる：

  1. 可能な限り、時間無制限。
  2. 学習障害や認知障害があるユーザーが回避しやすいタイムアウト。例えば、タスクの開始時に、ユーザーはタスクを完了するまでに丸1日あることを知らされます。これより長い時間に設定するオプションもあります。
  3. タイムアウトが必要な場合：
    1. タスクの開始時に出すタイムアウトの警告。 例えば、オークションでは、ユーザーが入札を行うための時間制限がある。タスクの開始時に、制限時間と終了までの時間について警告を出す。
    2. 可能かつ安全な場合は、ユーザーの作業内容は失われない。 例えば、機密情報を扱うWebサイトでは、コンピュータから離れる可能性のあるユーザーを保護するために、クライアントサイドの時間制限を使用する。ユーザーが再びログインしたとき、すべての作業内容がそのまま残っている。ユーザーは、非アクティブセッションをどれだけの時間持っていられるかについて事前に警告され、データが失われないことを告げられる。

避ける：

  1. セキュリティ上必要のないシステムタイムアウト。
  2. 該当プロセスの開始時にユーザーに警告しないシステムタイムアウト。
  3. セキュリティやプライバシーを脅かすことなく、ユーザーが作業内容を失うようなシステムタイムアウト。

フィードバックを提供する［パターン］

ユーザーニーズ

イベントが正常にトリガーされたことを示す、迅速なフィードバックや視覚的な合図が欲しい。例えば、いつメールが送信されたかを知る必要がある。そうでなければ、まるでメールが消えてしまったかのように感じる。

関連するユーザーストーリー： 3.4.1 アシストとサポート

やるべきこと

プロセス中の各ステップについて、そのステータスと正常に完了したかどうかをユーザーに知らせる。

メリット
ユーザーの各アクションの結果を明確にすることは、さまざまな認知障害や学習障害がある人たちの助けとなります。これには以下が含まれます
  • 自分のアクションが処理されたことを理解する（例：クリックして何かをした）。
  • 結果に対する不安や疑念を防ぐ。
  • 今何をしたかを覚えている。

例えば、加齢による物忘れがあるユーザーは、インターフェースがどのように動作したかを覚えているのが困難なことがあります。そのため、［送信］ボタンを押しても、フォームが送信されたことに自信が持てないかもしれません。そこで、「ありがとうございました」というメッセージなどのフィードバックがあれば、送信されたことが分かり、プロセスに自信を持てます。

また、複数のステップがあるタスクの場合、このフィードバック（ユーザーアクションへのフィードバック）は、注意力の障害や短期的な認知障害や学習障害がある人が、自分が何をしているのかを思い出すのに役立ちます。例えば、初期段階の認知症のユーザーは、気が散ってしまい、自分がタスクのどこにいたのかを忘れてしまうことがあります。このような場合、ユーザーアクションへのフィードバックにより、状況を再確認することができます。また、自分がプロセスの中にいること、そして現在プロセスのどこにいるのかを思い出させることによって、タスクからの離脱を回避できます。

ユーザーアクションの1つひとつに対して、容易に認識できる成功または失敗のフィードバックを提供する。可能であれば、フィードバックには一貫性のある慣れ親しんだデザインパターンを使用する。例えば：

  • 複数のステップがあるタスク内で、あるステップが完了すると、パンくずリストはそのステップの名前の横にチェックマークを表示する。該当する場合は、次のステップのタイトルや名前がすぐに分かるようにする。
  • ボタンがクリックされた後、ボタンが凹んでいるように見える。(トグルボタンの場合は、その状態はプログラム的にも特定できること）。
  • フォームの送信やメールの送信後は、「応募が完了しました、ありがとうございました」、「メールが送信されました」など、今何が起こったかを伝えるフィードバックを提供する。
詳細

ユーザが開始したすべてのアクションの成否は、コンテンツの主要モダリティにおいて、視覚的で、プログラム的に特定可能な、迅速なフィードバックによってユーザーに明確に示します。音声によるフィードバックもサポートされています。

事例

用いる：

  1. 視覚的な状態（state）のフィードバック。CSS および WAI-ARIA [[wai-aria-1.2]] の aria-pressed属性や aria-selected属性のような状態を使用して、状態の変化を視覚的に表現する。例えば、選択されたときの状態が視覚的に明確なボタンやタブ。
  2. タスクが完了したことをユーザーに知らせるメッセージ。例えば：
    • メール送信に成功したとき、またはフォーム送信に成功したときの確認メッセージ。
    • 新しいパスワードが設定されたことを示す、可視的でプログラムにより決定可能な情報。

避ける：

  1. 視覚的なフィードバックがない状態の変化。
  2. フィードバックがないタスクの成功または失敗。

ユーザーの安全を確保する［パターン］

ユーザーニーズ

Webサイトを使用する際、特に情報を提供したり他者とコミュニケーションを取る場合には、自分が安全でセキュアであることを確認したい。

関連するユーザーストーリー： 3.7.4 認知的ストレス

やるべきこと

ユーザーの安全を守る。これには以下が含まれます：

  • 認知障害や学習障害がある人が個人情報を提供したり、他の人とコミュニケーションを取ったりする際のリスクを理解する。
  • 高齢のユーザーや認知障害や学習障害があるユーザーなど、広範囲のカスタマイズされたプロファイルに対して、安全およびセキュリティのテクニックがどのように機能するかを確認する。
  • 機密性の高いユーザー情報を安全に保つために既知のテクニックを使用する。
  • すべてのユーザーが関連する既知のリスクを理解できるようにする。あらゆる既知のリスクについて、理解しやすく親しみやすい言葉で説明する。
  • これにより、ユーザーが十分な情報を得た上で意思決定を行い、コントロールし続けることができる。
メリット

ユーザーは、ウェブサイトを利用する際、特に情報提供や他者とのコミュニケーションを行う際に、自分が安全でセキュリティが確保されていることを知る必要があります。

遂行機能が低下しているユーザーは、リスクを正しく認識することが難しいため、潜在的なリスクを明確にすることによって、ユーザーが安全でコントロールしやすい状態を維持することができます。コンテンツを安全に利用するために役立つヒントを追加し、問題が発生した場合のヘルプを提供してください。

リスクを特定するために、認知障害や学習障害がある人たちと調査やフォーカスグループを実施し、認知障害や学習障害のある人たちと協力して、潜在的な問題や既存の問題を解決することをお勧めします。グループは、セキュリティとリスク軽減に取り組む際に、学習障害や認知障害のある人々を念頭に置くべきです。

例えば、パスワードのコピー＆ペーストや二段階認証コードの使用ができない人の多くは、介護者に助けを求めています。介護者は派遣社員であることが多いため、この場合、ユーザーは無防備な状態になってしまいます。パスワードを長くしたり、定期的に変更することを義務付けると、こうした安全でない習慣が増え、実際にアプリケーションの安全性が損なわれる可能性があります。このような設計ミスは、認知障害や学習障害がある人がユーザー調査や分析から外されている場合によく起こります。

事例

用いる：

  1. 管轄区域において機密データ用にセキュリティ技術として承認されている、学習障害または認知障害がある人々によってテストされた代替ログインのオプション。
  2. 広範囲の学習障害や認知障害がある人々に話を聞き、そのリスクと個人情報との関連性を理解する。
  3. 研究、開発、要件の段階におけるCOGAペルソナとユースケース。
  4. ユーザー情報の保存と保護に関する業界のベストプラクティス。
  5. 認知障害や学習障害がある人がリスクを理解できることを検証した、理解しやすい言葉で書かれた同意書。
  6. 個人情報が提供される可能性がある場合、その都度、理解しやすい言葉で書かれたユーザーへの警告。

避ける：

  1. ユーザーがすべてのリスクを理解せずに情報を共有している。
  2. リスクに関する情報が隠され、混乱している。
  3. ユーザーは一回同意しただけで、現在直面しているリスクについて忘れてしまっている。
  4. ユーザーが理解できない可能性がある同意書。

馴染みのある測定基準と単位を使う［パターン］

ユーザーニーズ

私が知っていて、私の居場所で一般的な測定基準 (フィートやメートルなど) を使用するインターフェースにしてほしい。そうでないと、混乱してしまう。私は常に、相手がどの測定基準について話しているのかを知っているわけではないし、数字が間違っているように見えても気付かない。

関連するユーザーストーリー： 3.4.1 アシストとサポート

やるべきこと

ユーザーが慣れ親しんでいる単位で測定基準を提供する。

メリット

ほとんどの人は、自分の住んでいる場所や文化の中で、測定基準によく使われる1組の単位に慣れています。測定基準が他の単位である場合、彼らはそれを理解するために変換する必要があります。電卓のようなツールでさえ、対処するのが難しい場合があります。単位を変更するオプションを提供し、ユーザーの所在地に応じた単位をデフォルトにしてください。よくある例としては、距離、重量、面積、通貨、温度などに使われる単位があります。

例えば、あるユーザーが摂氏で温度を知っているとします。華氏で表示されると、とても暖かくなると思うでしょう。

詳細

ローカライズされた代替の単位が利用可能であっても、特定の単位で測定基準が用いられていることがよくあります。例えば、テレビやモニターのサイズは、センチメートルが一般的な単位であっても、通常はインチで表示されます。しかし、このような場合でも、ユーザがその測定基準に慣れていない可能性があるため、代替手段を提供することは有用です。

まずはじめに

ユーザーにとってより意味のある別の測定基準セットを選択する仕組みを提供する、または本文中に一般的な代替を提供する。

事例

用いる：

  1. 様々なユーザーが理解する別の単位での測定基準。例えば：
    • エッフェル塔の高さは、頂上のアンテナを含めて、1,063フィート（324メートル）です。

避ける：

  1. 測定基準を1つの単位で表すこと。例えば：
    • エッフェル塔の高さは、1,063フィートです。

目標 5: ユーザーが集中できるようにする。

注意力が散漫になると、認知障害や学習障害があるユーザーは、タスクを完了できなくなることがあります。

ユーザーは一旦気が散ってしまうと、自分が何をしていたかを思い出すのが難しくなることがあります。そうなると、もはやタスクを完了することはできません。これは、認知症のユーザーなど、注意力と記憶力の両方が低下しているユーザーにとって、特に問題となります。

ユーザーの注意をそらしたり、邪魔になるようなコンテンツや要素の使用は避けてください。また、ユーザーが不要と感じるコンテンツは削除することを検討してください。ユーザーが集中力を欠いたときに状況確認や再集中ができるように、明確な見出しやパンくずリストを提供してください。

また、タスクを開始する前に必要な情報をすべて集めておけるように、タスクの最初に、準備する必要がありそうな情報をユーザーに伝えて、ユーザーがタスクに集中できるようにしてください。

割り込みを制限する［パターン］

ユーザーニーズ

タスクは気が散らないようにしてほしい。

関連するユーザーストーリー： 3.5.1 注意散漫

やるべきこと

割り込みを避ける。これには以下が含まれます：

  • ユーザーが開始した場合、または緊急の場合を除き、割り込み、リマインダー、およびコンテンツの変更を制御する簡単な方法を提供する
  • ユーザーが注意散漫になったり、好ましくない反応をする可能性があるコンテンツを制御し、制限できるようにする。

このコンテンツには以下が含まれます：ソーシャルメディア、暴力的なコンテンツ、広告、気が散る背景や画像、動くコンテンツ、小さな音や大きな音、トリガーなど。

メリット

割り込みは、記憶や注意に障害のある人がタスクを完了するのを妨げます。これには、認知症の人、脳卒中や脳損傷がある人、記憶や注意に影響を与える副作用のある薬を服用している人などが含まれます。特定の種類の割り込みや特定の数の割り込みは、たとえそのタスクが非常に重要であっても、作業を中断させることがあります。割り込みには、音、視覚的に現れたり変化したりするコンテンツ（ページ上の広告など）が含まれます。割り込みは、共有のオンラインドキュメントで作業しているときに、新しい変更の存在をテキストで通知するような単純な場合もありえます。

記憶力や注意力に問題がある人にとっては、以下のようなサイトが最も良く機能します：

  • 静かでシンプルな環境である。
  • 割り込みが一切ない。
  • 割り込みや移動したコンテンツを後で見ることができるように、簡単に使える［一時停止］のオプションがある。
  • 設定で管理する割り込みの種類やタイミングをユーザーが選択できる。

天気の悪化による休校など、重要な情報を読み取る必要がある場合、多くのニュースサイトでは、多くの中断が発生します。ニュース速報のテキスト、広告、ポップアップウィンドウが表示されることもあります。集中力が続かず、学校名を確認するのが難しい人や、2つ3つ確認したい人がいる場合、これらの邪魔が入ることで作業が不可能になる可能性があります。このような邪魔になるものを一時停止させ、理想的にはページから一時的に取り除くことで、ユーザーはタスクを完了できるようになります。

人によってはノイズに敏感で、多すぎる刺激に圧倒されてしまうことがあります。

時には、ノイズや様々な種類のコンテンツが、メンタルヘルスに悪影響を及ぼすことがあります。例えば、ノイズ、邪魔になるもの、苦痛を与えるコンテンツは、ユーザーをより不安にさせ、心的外傷後ストレス障害（PTSD）を誘発する可能性があります。また、あまりに多くの割り込みやソーシャルメディアの使用が、うつ病や集中力の欠如を悪化させる可能性があることを示唆する研究もなされています。このようなコンテンツをユーザーがコントロールできるようにすることで、オンラインでの生産性を高めることができるかもしれません。

邪魔になるものを取り除いたり、コントロールするための標準的な手法が存在する場合は、それを使用するべきです。

例えば、外傷性の脳損傷がある人がオンラインで税金の申告をしているとします。その際、ソーシャル・メディア・アプリケーションが通知を表示します。通知をオフにしてから、アプリケーションをオフにしようとしますが、複雑すぎます。手助けがなければ税金を納めることができません。

事例

用いる：

  1. 割り込みのないタスク。
  2. 必要と思われる割り込みを簡単に制御する方法。例えば：
    • あるアプリケーションでは、リマインダーや電子メールの通知方法をユーザーが決めることができます。ユーザーは、視覚的なリマインダー、音、または何もしないことを選択することができます。また、ほとんどの場合において割り込みを許可するユーザーを重要な連絡先としてフラグを立てることができます。これらの設定は、どの画面からも簡単に見つけることができます。中には、タスクに集中して、完了したらメールやカレンダーを見るようにするために、何も通知させないという使い方をするユーザーもいます。

避ける：

  1. オフにするのが難しい、または消すことができない邪魔になるコンテンツ。例えば：
    • 雑誌の記事ページに、常に変化して読者の集中を妨げる広告がある。
  2. ユーザーのタスク完了を妨げるコンテンツ。例えば：
    • ユーザーにサイトの購読を勧めるポップアップ。ユーザーが作業を続けるには、このポップアップを閉じなければならない。

クリティカルパスを短くする［パターン］

ユーザーニーズ

機能やコンテンツを簡単に見つけたい。

関連するユーザーストーリー： 3.2.2 検索可能

やるべきこと

プロセスやワークフローを合理化し、最低限必要なステップのみを含めるようにする。補足的であるが必須ではないオプションのステップを切り離す。ユーザーにオプションのステップを経ることを要求しない。

メリット

プロセスやワークフローを合理化することによって、注意力散漫、ミス、精神的疲労を軽減することができます。短いクリティカルパスを用いると、認知障害や学習障害があるユーザーがプロセスやタスク、ワークフローを正常かつ正確に完了できる可能性を高めることができます。

例えば、初期段階の認知症のユーザーが新しい携帯電話を買おうとしているとします。しかし、支払いを済ませる前に、ヘッドフォンやその他のアイテムを提案するステップが追加されます。ユーザーは圧倒され、混乱します。結果として、携帯電話を買わずにサイトを出て行ってしまいます。

事例

用いる：

  1. 必要なステップだけを要求するプロセス。
    • 例えば、 映画のチケットを購入するためのオンラインプロセスに含まれる手順は、以下の通りです：
      • 映画を選択する。
      • 日付と時間を選択する。
      • 座席を選ぶ。
      • 料金を支払う。
      • チケットを印刷して保存する。

      映画館では、ユーザーが映画に関する説明や評価を見たり、事前にスナックを購入したり、チャリティーに寄付したりすることができます。これらのアクション（ステップ）は、ユーザーが映画のチケットを購入するというタスクを完了するために必要なものではありません。購入プロセスの一部としてユーザーにそのような選択をさせるのではなく、購入手続き開始前と完了後に選択肢を提示します。

  2. 主要な機能への簡潔なナビゲーション。例えば、あるアプリで最もよく使われる機能の場合：
    • アプリを開く。
    • 最もよく使う機能を実行する。

避ける：

  1. 不要なステップの追加。例えば、映画のチケットを購入するためのオンラインプロセスに含まれるステップは、以下の通りです：
    • 映画を選択する。
    • 日付と時間を選択する。
    • 座席を選択する。
    • スナック菓子の購入、またはオプトアウト。
    • チャリティーの寄付をする、またはしない。
    • アカウントを作成する。
    • 支払う。
    • チケットを印刷する。

映画館では、チケット代金を支払う前に、軽食を決め、チャリティ寄付をするようユーザーに強制しています。オプトアウトのオプションも用意されていますが、画面上、特にモバイル機器ではやや隠れてしまい、ユーザーは支払い方法がわからずに立ち止まってしまうことが多いようです。

  1. 最もよく使う機能のための冗長なナビゲーション。例えば、あるアプリで最もよく使う機能の場合：
    • アプリケーションの紹介ページに移動し、［続ける］を押す。
    • アプリケーションのメインページに移動する。
    • サブページへ移動する。
    • オプションを選択する。
    • 別のオプションを選択する。
    • 最もよく使う機能を実行する。

コンテンツ量をあまりにも多くならないようにする［パターン］

ユーザーニーズ

必要なコンテンツと不要なコンテンツを簡単に識別したい。私が知る必要のある情報や重要な情報は目立ち、または私が最初に読むものであり、重要性の低い情報のノイズの中に紛れ込むことはない。

関連するユーザーストーリー： 3.2.1 探索可能

やるべきこと

インターフェイスはシンプルに保つ。各画面で5つ以下の主な選択肢をユーザーに提供し、不要なコンテンツは削除する。これは、メインコンテンツと同じコードベースからリアルタイムに生成される代替案として、簡易版で提供することができます。

ページの主目的と関係のない余分なリンクは、フッターセクションに限定すべきです。余分な選択肢は、「もっと見る」リンクや他の明確で説明的なタイトルの下に隠しておくこともできます。

メリット

コンテンツが多いページ、多すぎるテキスト、多すぎる画像、その他の多すぎるコンテンツは、認知の過負荷、不安、そして集中力の欠如を引き起こします。コンテンツを少数の重要なポイントに絞ることによって、混乱を減らし、ユーザーを落ち着かせ、そして記憶を助けながらより良く理解することを可能にします。例えば、複雑そうに見えるとページを離れてしまうような、読むのが遅い人や注意力の低い人にも有効です。

シンプルなコンテンツと一貫したシンプルなデザインは、認知の過負荷を減らし、ストレスや精神的疲労を減少させるのに役立ちます。例えば、初期段階の認知症の人が医師の診療申込書を取りに行ったとします。画面には、予約、医師への質問、検査結果、承認など、5つの選択肢が表示されています。各選択肢にはアイコンがあり、明確なテキストがあり、空白で区切られています。2回のクリックで、医師に質問をすることができました。助けを求めることなく、必要なものを簡単に選択することができます。左にスワイプすれば、さらに多くの選択肢を見ることができます。しかし、そうすることはまずないでしょう。

詳細

長い段落、たくさんの選択肢、意味のない画像などを避けることによって、認知障害や学習障害がある人が重要なポイントに集中できるようにします。

いくつかの短い箇条書きだけにして、Webサイトやサービスの主な主題に関連する画像を1つか2つに限定することで、ユーザーはそのサイトをさらに探索するかどうかを選択することができます。

このパターンの意図は、不要な情報でページを乱雑にすることではなく、認知障害や学習障害がある人にとって重要な手がかりや説明を提供することです。情報や説明が多すぎるのも、少なすぎるのと同じように、障害になる可能性があります。目標は、過度の混乱やナビゲーションなしで、ユーザーがタスクを達成できるように、十分な情報を提供することです。

事例

用いる：

  1. シンプルなインターフェース。主な機能は、他のどれよりも大きい。例えば：
    • 検索エンジンは、その名前と大きくてシンプルな検索ボックスがある。他のコンテンツはすべて小さくて下の方にあり、ユーザーは追加の機能を探さないかぎり、それに気づかない。

避ける：

  1. 不要なコンテンツがあるページ。例えば：
    • 多すぎるテキスト、長いメニュー、ぎっしり詰まったテキストの長い段落を囲むように配置された画像。情報過多で、メッセージが失われている。

ユーザーが事前に準備してタスクを完了できるように情報を提供する［パターン］

ユーザーニーズ

タスクを開始する方法と、タスクの内容を知りたい。

関連するユーザーストーリー： 3.5.1 注意散漫

やるべきこと

重要なタスクの開始を強調する。

ユーザーが複数のステップからなるタスクを実行する前に、そのタスクを完了するために必要な労力を見積もれるようにする：

  • かかるであろう時間
  • タスクを実行するために必要なリソースの詳細
  • プロセスや次のステップの概要

ユーザーがタスクを開始したら、そのタスクがまだ「進行中」であることや、完了していることを明確に理解できるようにする。

メリット

一部のユーザーは、注意散漫になることによってタスクの途中でフォーカス対象が切り替わり、その後中断したところに戻るときに、注意散漫になることが難題だと感じています。例えば、あるウェブサイトでは、「予約はこちら」というリンクへの方向を示す大きな矢印が表示されます。この矢印は、予約タスクの開始を強調し、ユーザーがタスクを開始したことを認識するのに役立ちます。

多くの場合、人は集中する時間を管理し、中断されることなく集中できるようにする必要があります。タスクにかかる時間、複雑さ、ワーキングメモリの負荷について事前にアドバイスを受けることにより、より良い準備と完了が可能になります。タスクを開始する前に必要なリソースをリストアップし、タスク完了までの残りステップ数を表示することで、ユーザーが失敗を回避するのに役立ちます。

まずはじめに

複数ステップのタスクやフォームの開始時に、必要な時間の見積もりと必要なすべてのリソースのリストを提供する。また、タスクをステップに分割する。

事例

用いる：

  1. ユーザーがタスクを開始する際の視覚的な手がかり。
  2. 作業にかかる時間や必要なリソースなど、プロセスの概要。例えば：
    • ユーザーが航空券の予約を開始する前に、「航空券の予約にかかる平均時間は15～30分ですこのプロセスを完了するには、旅行の日程、旅行者の数、旅行者のパスポートが必要です。」というメッセージを表示する。

避ける：

  1. ユーザーが必要とする可能性があるものについての重要な詳細が欠落していること。例えば：
    • 別の航空会社は、ユーザーにパスポートが必要であることを通知していない。ユーザーがパスポート番号を探しているときに、プロセスがタイムアウトする。ユーザーは予約をやり直すか、予約を諦めてしまう。

目標 6: プロセスが記憶に依存しないようにする。

記憶のバリアにより、多くのユーザーが製品を使用したり、ヘルプやコンテンツにアクセスしたりできなくなります。

記憶や言語に何らかの障害がある人は、記憶のバリアを克服することが困難であったり、不可能であったりする場合があります。

例えば、多くのユーザーは短期記憶に障害があります。平均して、人は同時に7つの文字や項目を記憶することができます。ワーキングメモリーに障害のある人は、同時に1つから4つの情報を記憶することができる可能性があります（障害の程度によって異なります）。もし、自分が何をしたかを記録するなど、他のタスクも覚えておく必要がある場合は、ミスをする可能性が高くなります。

以下のようなバリアを回避してください。

  • 特定の数字や用語を覚えなければならない音声メニューを操作する。
  • 音声メニューの単語を処理しながら、数字を覚える。
  • テキストを書き写す
  • パスワードを記憶する

ユーザーがコンテンツ、サービス、またはヘルプにアクセスできるようにするには、記憶に頼るプロセスを使用しないようにします。必要な人には、より簡単なオプションを提供してください。

記憶やその他の認知スキルに依存しないログイン方法を提供する［パターン］

ユーザーニーズ

パスワードやユーザー名を覚えたり書き写したりせずに、サイトを利用したい。

関連するユーザーストーリー： 3.6.2 アクセシブルな認証

やるべきこと

ユーザーは、シンプルなウェブページを利用するのに必要な認知能力があれば、ログイン、登録、認証情報のリセットを行うことができる。ユーザーは、以下のことをする必要はない：

  • 文字列を記憶する。
  • 計算をする。
  • コンテンツをコピーする。
  • パズルに答える。
  • ジェスチャーを確実に再現する。
  • 画面に表示された文字を認識し、入力フィールドに入力する。
メリット

記憶障害がある人は、パスワードを忘れてしまい、ログインできないことがよくあります。その解決策は、役に立つときもありますが、セキュリティ上のリスクがあることも多いのです：

  • テキストをコピーしたり、フォームに入力するために、何度もテキストを見たり聞いたりしなければならない場合がある。
  • 覚えておくことが出来るように、1つのパスワードを再利用したり、覚えやすいパスワードを使っている。
  • パスワードを変更したり、複雑なパスワードを使用したりする必要がある場合、他の人が見ることができる紙片に書くなどして、パスワードを安全に保管できないことがあります。

また、ログインするときに、以下のようなステップで苦労することもあります：

  • 文字を正しい順序で入力する
  • 限られた回数で正しく文字を入力する（ロックアウトされる）。
  • 暗証番号を確認する。
  • パズルや歪んだ文字を読み解く。
  • 他のデバイスから文字を正しくコピーする。

ユーザーは、時間制限のある手順やデジタルセキュリティトークンの提示に挫折してやめてしまうことがあります。また、障害があるために重要なサービスから締め出されることもあります。

この設計要件がなければ、多くの人がアプリケーションやコンテンツをまったく利用できないことになります。この問題の詳細については、「Security and Privacy Technologies」の論点書をご覧ください。また、この問題が、しばしば重要な意味を持つウェブサービスの利用をいかに妨げているかについても説明しています。例えば、多くの人は医者の予約などを自分で取ることができません。このことは、認知障害や学習障害がある人々の寿命が短くなる原因の一つかもしれません。

詳細

このデザインパターンを満たすには、様々な方法があります：

  • Web Authentication: An API for accessing Public Key Credentials [[webauthn-2]] を使用して、一般的な認知能力に依存しないインクルーシブな代替手段をサポートする。例えば、ハードウェア認証デバイス、スマートカード、FIDO など。
  • 信頼できるデバイス（ユーザーが既に自分の ID でログインしている）を使用していることに基づいた、自動的なユーザー認証を提供する。ユーザーは文字を書き写す必要はないが、そのデバイスを持っていることを確認するためにリンクを押す必要がある場合がある。
  • 生体認証オプションを許可する。
  • サードパーティの認証サービス（OAuth など）により、ユーザーが既にログインしているシステムを使用する。

代替ユーザー認証の要件を満たす方法としては、以下のようなものが考えられます：

  • メールアドレスや電話番号に送られたリンクをクリックする。(これは実装が簡単で、ブログのコメントを許可するような最小限のセキュリティに有用であることに注意してください)。
  • 現在の口座残高の合計数など、ユーザーの個人文書に存在する情報を使用してログインする。なお、この情報を見つける方法についての説明を添える。
事例

用いる：

  1. Web Authentication: An API for accessing Public Key Credentials [[webauthn-2]].
  2. ユーザーが1回のログイン(フェデレーション認証)で多くのサイトにアクセスできるようにするシングルサインオン(SSO)。
  3. Bluetoothリンクによる2段階認証 (コピーなし)。
  4. クイックレスポンスコード (QRコード)。

避ける：

  1. コピーを必要とする2段階認証の使用。
  2. パスワードを使用し、フィールドへの貼り付けを許可しない。

ユーザーがシンプルにシングルステップでログインできるようにする［パターン］

ログインの手順を複数のステップにしないで、シンプルにしてほしい。

関連するユーザーストーリー： 3.6.2 アクセシブルな認証

やるべきこと

ログインに、シンプルでシングルステップの代替手段を提供する。

メリット

シンプルなログインにより、遂行機能や記憶力が低下している人でもアプリケーションを使用することができます。これは、複数のステップからなるプロセスで混乱したり圧倒されたりするユーザーにとって特に重要です。例えば、外傷性の脳損傷があるユーザーがオンラインバンキングのサイトを利用したいとします。指紋スキャナーに指を置いて、本人であることを認証できるかもしれません。他の例としては、サードパーティーのログインがあります。

事例

用いる：

  1. オプションとして、サードパーティの簡単なログイン。
  2. セキュリティニーズにマッチするシングルステップ方式を採用したWeb認証プロトコル [[webauthn-2]]

避ける：

  1. 複数のステップがあるログイン方法全て。

より少ない文字数でログイン可能な代替手段を提供する［パターン］

ユーザーニーズ

（重度の言語障害があるため）多くの文字数を必要としない、自分が使えるログインプロセスを使いたい。

関連するユーザーストーリー： 3.6.2 アクセシブルな認証

やるべきこと

多くの文字を読んだり書いたりする必要のないログイン方法を少なくとも1つ提供する。

メリット

このパターンにより、言語やコミュニケーションに障害がある人でも、テキストのブロックに圧倒されることなく、ログインすることができます。

例えば、AAC（代替・拡張コミュニケーションシステム）デバイスを使用している重度の言語障害がある人が、医師にメッセージを送りたいとします。その場合、テキストを読むことなく、自分の知っているアイコンの［ログイン］ボタンを押して、メッセージを送信することができます。

事例

用いる：

  1. よく知られたアイコンを用いたサードパーティのログイン。
  2. Web認証プロトコル[[webauthn-2]]に沿った簡単なログインオプション。

避ける：

  1. セキュリティに関する質問への回答を要求するログイン。
  2. 簡単で言葉は不要のオプションや代替手段がないログイン。

ユーザーが音声メニューを操作しなくて済むようにする（パターン）

ユーザーニーズ

複雑なメニューシステムを介さずに、人間のヘルプを利用したい。

関連するユーザーストーリー： 3.6.3 音声メニュー

やるべきこと

ヘルプできる人に容易に連絡できるようにする。担当者にコンタクトするために、メニューシステムの使用を求めない。

音声メニューは以下のように設計する：

  • 音声メニューをスキップして直接人にコンタクトするために、いつでも使える既知の単語または予約番号（「0」または「ヘルプ」など）を提供する。
  • 不必要なステップやオプションを避ける。
  • プロモーション情報など、不要な情報や注意をそらすような情報を避ける。
  • さまざまな認知障害や学習障害がある人が普通に使う言葉を使う。
  • 遅い話し手が反応するのを待つ。
  • 静かな話し手、繰り返し、吃音を許容する。
  • 物忘れや記憶障害をサポートする。
  • エラーの回復を容易にする
  • ユーザビリティのベストプラクティスに従うこと。
メリット

音声メニューが使えない人は多い。そのため、重要なタスクを自分で完了できないことがよくあります。例えば、医者の予約、健康保険の加入、ソーシャルサービスの利用、水道の復旧などです。

音声メニューを自分で操作できない場合、誰かに助けを求めなければなりません。これでは、サービスはアクセシブルではありません（利用できません）。タスクが完了しないこともあります。例えば、ヘルパーの手を煩わせないようにと、医師の予約やその他の重要な作業を遅らせてしまうことがあります。必要な支援が受けられない、あるいは手遅れになることが多いのです。(認知障害や学習障害がある人の平均寿命が短いのは、このことが一因かもしれません）。

なぜ人は複雑なメニューを使えないのか？

  • ユーザーがメニューの番号や用語を覚えるためには、優れた短期記憶と作業記憶（数秒）が不可欠です。これらの記憶がなければ、ユーザーは間違った番号を選択する可能性が高くなります。
  • 多くのユーザーはワーキングメモリーの機能が低下しています。例えば、あなたが7つの文字や項目を同時に頭の中で記憶できたとしても、人によっては1つか2つしか記憶できないかもしれません。そのため、メニューシステムを正しく管理することが難しくなります。
  • 例えば、電話のメニューシステム（音声マークアップ・システム）には、「3を押して、社内サービスへ」というようなオプションがあります。数字が先に出てくると、社内サービスが必要かどうかを判断する前に、ユーザーは3という数字を覚えなければなりません。多くの人はこれを行うことができません。また、正しい数字を押すことも必要です。
  • 正しい選択肢の前に多くの無関係な情報が与えられると、ユーザーは認知的過負荷に陥り、立ち止まってしまうかもしれません。特に、以前の選択肢や情報をすべて理解していなかった場合は、そうなります。
  • 使用されている用語を理解していない可能性があります。メニューに集中した結果、なぜ電話をしているのか忘れてしまうこともあります。
  • ウェブメニューのような代替ルートが提供されることもあるが、それも複雑である。そのため、役に立ちません。

0桁の数字、「ヘルプ」、またはその他の文化的に適切な用語は、人にコンタクトするときに使うとよいでしょう。各メニューの最初の選択肢を一貫して「サポートできる人を待つには、0を押してください」にしてください。これにより、誰もが必要なサポートにたどり着くことができます。

Tあらゆるシステムを認知障害や学習障害がある様々なユーザーでテストしてください。様々なユーザーに参加してもらうことによって、多くの音声システムが、認知障害や学習障害のある人の助けとなる程度までには、アクセシブルになってきました。

詳細

一般的な音声ユーザーインターフェース（VUI）設計のベストプラクティスに従うこと。音声ユーザーインターフェースの標準的なベストプラクティスは、認知障害があるユーザーにも適用されるため、それに従う必要があります。対話型インターフェースに焦点を当てた設計が増えるにつれ、これらのプラクティスは改善されています。音声ユーザーインターフェースの設計で一般に受け入れられているベストプラクティスの例をいくつか挙げます：

  • 言語とオプションの処理時間を確保するために、フレーズとフレーズの間に一呼吸を入れることが重要である。
  • 選択肢の内容は、選択する数字や、その選択肢を選択する指示の前に提示する。そうすることで、ユーザーはその言葉を処理する際に、数字や指示を覚えておく必要がなくなる。例えば、プロンプト「1 を押して、秘書を呼ぶ」では、ユーザーは「秘書」という用語を解釈するまでの間、数字の「1」を覚えておく必要がある。より良いプロンプトは「秘書を呼ぶ（一時停止）：1を押してください」または「秘書を呼ぶ（一時停止）またはヘルプが必要なら（一時停止）：1を押してください」となる。
  • エラーの修正は容易に出来るべきであり、エラーが続く場合は、ユーザーを人間のオペレーターのところに案内すべきである。エラーへの対応は、通話を終了させたり、ユーザーをより複雑なメニューに案内したりしてはいけない。
  • 広告やその他の余計な情報は、ユーザーを混乱させ、注意力を持続させることが難しくなるため、読まないようにする。
  • 使用する用語は、できるだけシンプルかつ専門用語でないものにする
  • 先細になるプロンプトを使用する。音声ユーザーインターフェースの設計におけるベストプラクティスとして、インタラクションの各ポイントに対して複数の様々なプロンプトを提供することが挙げられます。様々なプロンプトは、ユーザーの行動に基づいて使用されます。例えば、ユーザーがプロンプトに応答するのに長い時間がかかる場合、デフォルトのプロンプトの代わりに、よりシンプルなプロンプトやより説明的なプロンプトを使用することができます。これらは、ユーザーが期待通りに反応しない場合に、プロンプトの詳細レベルを上げるために使用する必要があります。

ユーザーの設定

ユーザー固有の設定を使用して、音声ユーザー インターフェイス (メニューやオプションなど) をカスタマイズできる。ユーザープリファレンスの設定が難しい場合、それらは使用されないことに注意する。自然言語によるプリファレンスの設定 (「速度を落としてください!」)が最も自然ですが、現在はあまり一般的ではない。

  • 音声のスピード及びゆっくり話すか、入力時間を長くするかは、ユーザーが設定できるようにする必要がある。時間制限のあるテキストは、（すべてのアクセシブルなメディアと同様に）調整可能にすべきである。
  • タイムアウトは、ユーザーが自分のデバイスのシステムデフォルトとして延長または無効化できる必要がある。
  • エラーの修正はシンプルであり、また人間のオペレーターのところに案内すべきである。エラーの対応は、ユーザーを回線から切り離したり、より複雑なメニューに案内したりするものであってはならない。サポートと人間によるヘルプのために予約番号を使用することが好ましい。
  • 広告やその他の情報は、ユーザーを混乱させ、注意力を持続させることが難しくなるため、読まないようにする。
  • 使用する用語はできるだけシンプルにする。
  • 認知的負荷を軽減するプロンプトの構築方法について、事例とアドバイスを提供すべきである。
    • 例1．認知的負荷を軽減する プロンプト「秘書を呼ぶには1を押す」では、ユーザーは秘書という用語を解釈するまでの間、数字の1を覚えておく必要がある。これは、プロンプト「秘書を呼ぶ（一時停止）：1を押してください」や「秘書を呼ぶ（一時停止）またはヘルプが必要なら（一時停止）：1を押してください」よりも劣る。
    • 例2: デフォルトを人間のオペレーターにして、数字0を設定する。

音声認識に関する考察

  • 音声認識をベースにしたシステムには、多くの言語に対応した音声コマンドの規格があり、ETSI規格のETSI 202 076など、可能な限り使用することよい。数個以上のコマンドを覚えさせるのは、ユーザーにとって認知的な負担となることを念頭に置くこと。
  • 自然言語理解システムは、ユーザーが自分の言葉でリクエストを口述することを可能にする。これは、メニューオプションを覚えるのが困難なユーザーや、提示されたメニューオプションを自分の目的に対応させるのが困難なユーザーにとって有用である。しかし、自然言語インターフェースは、音声や言語を発することが困難なユーザーにとって使いにくい場合がある。そのため、直接対話（メニューベース）のフォールバックやエージェントへの転送を提供すべきである。このようなフォールバックは使いやすく、この文書に準拠している必要がある。

法規制の要件に従う。

例えば、米国電気通信法第255条アクセシビリティガイドライン [[Section255]] 1193.41項 入力、制御、機械的機能(g), (h) 及び (i)が認知障害に適用されます。時間に依存した制御、発話能力がなくても機器が操作できること、及び認知能力が限られている人が操作できること、が要求されています。

詳細については、 Voice Menu Systems の論点書を参照のこと。

まずはじめに

健康、金融、通信、水、行政サービスに影響を与える重要なシステムにおいて、このパターンを用いる。

事例

用いる：

  • 最初の選択肢で人間に直接コンタクトできるユーザー対話ダイアログ。例えば 「あなたを助けられる人にコンタクトするには、0を押す。または、「ヘルプ」と言ってください」。
  • ユーザーは簡単にエラーを修正できる、各ポイントからのエラー回復。
  • ユーザーはどの時点でも簡単にリストに戻ったり、繰り返したりすることができる。
  • 任意のポイントから標準的な「0」のようなユーザーインタラクションダイアログがあり、ユーザーが目的を達成するのを助ける人間のオペレーターに簡単にアクセスできる。
  • 音声は明瞭で、間がある。
  • 一般的な単語が使用されている。
  • 押すべきキーや使用する用語を提示する前に、選択肢を述べる。
  • 推奨手法：後になってから役に立つかもしれない何かを記録するようユーザーに合図し、そのための時間を与えること。

避ける：

  • 人を見つけるのが大変な長いメニューシステム。
  • 不明確なエラー回復。
  • 複雑な言葉や一般的でない用語。
  • 宣伝などの不要なコンテンツ。
  • 早口、またはユーザーからの早い入力を期待するもの。
  • さまざまな認知障害や学習障害を持つ幅広いユーザーでテストされていないシステム。

ユーザーの計算や情報の記憶に頼らない（パターン）

ユーザーニーズ

記憶に頼らないナビゲーションやプロセスにしてほしい。

関連するユーザーストーリー： 3.6.1 前のステップ

やるべきこと

以下を必要としないプロセスを作る：

  • 短時間で選択するために数字を覚えておくこと。
  • 計算をする。
  • コピーする。
  • 明瞭な会話や素早い応答。
  • 文字、文字列、暗証番号の記憶。
  • 必要なサービスのカテゴリーを割り出すために遂行機能を使う。
  • 複数のステップを経て、情報を思い出すこと。複数のステップを経る場合、プロセス中の各ステップには、ユーザーが先に進むために必要な情報が含まれていなければならない。前のステップの記憶に頼ってはいけない。

説明文やラベルは、Call To Action（CTA：行動喚起）や起動するメカニズムの前に配置する。適切な場合には、前のステップの情報の要約と、プロセスを横断するメカニズムを提供すること。

メリット

認知障害や学習障害がある人にとって、コンテンツに障壁があると、ステップやプロセスを完了することができず、結果として、達成したいことが達成できないことがよくあります。

購入時のセキュリティを高めるために、パズルや計算を要求する場合があります。例えば、ユーザーは、ある数字の 1 桁目と 2 桁目を掛けた結果を入力するよう求められるとします。このような文章は、認知障害や学習障害のある人には理解できない可能性があります。セキュリティと認知的アクセシビリティの両方を保証すべきです。

多くのユーザーは、ワーキングメモリに障害があります。ユーザーは、同時に多くの詳細を記憶することができません。しかし、ダイアログメニューなどのシステムは、すべてのユーザーがワーキングメモリーを使用して選択することに依存しています。ユーザーが複数の選択肢を記憶し、発話するか、キーを押すかによって、1つの選択肢を選択することを期待しています。ユーザーは、複数の一時的な情報を頭の中に保持しておく必要があります。

遂行機能が低下しているユーザーは、タスクを完了するために、より多くの時間を必要とするかもしれません。しかし、システムの応答が遅すぎる場合も問題があります。例えば、「請求」「口座」「販売」など、似たような選択肢を比較して、どれが自分に必要なサービスかを判断するのに時間がかかる人もいます。分かりやすい言葉を使うことも効果的です。

詳細

記憶力への依存を減らすグッドプラクティスには、次のようなものがあります：

  • プロセスの各ステップには、ユーザーが処理を進めるために必要な情報が含まれている必要がある。 前のステップの記憶に頼ってはならない。
  • 最初からやり直すことなく、前に戻るための簡単な方法を提供する。
  • ユーザーがプロセス中のどの段階にいるのか、例えばステップインジケーターで知らせる。
  • ユーザーがミスやエラーを起こした場合、プロセスを再開するのではなく、それを修正するためのヘルプを提供する。
  • 有用な場合は、前のステップの情報の要約を提供し、プロセスを簡単に横断する仕組みを用意する。

音声インターフェースの場合：

  • 各オプションの間にポーズを入れる。
  • 遅い話し手が応答するのを待つ。
  • 静かな話し手や躊躇している話し手に耳を傾ける。
  • シンプルで一言の応答を許可する。
  • 選択するよう命令する番号の前に選択肢を提示する。
  • 物忘れや記憶障害に対応する。

なお、健康、金融、通信、水、行政サービスなどの重要なシステムには必須であることに留意してください。

事例

用いる：

メモリに依存しないプロセス。例えば：

  • 最初の選択肢が「助けてくれる人を待つには、0を押すか、ヘルプと言ってください。」であるユーザーインタラクションダイアログを使用する。
  • 標準的な「ヘルプ」のように、ユーザーが目標を達成するのを助けることができる人間のオペレーターに簡単にアクセス可能なユーザーインタラクションダイアログを使用する。
  • ユーザーがどの地点からでも戻る方法を知っている、優れたエラー回復。
  • 推奨方法：ユーザーに、後の時点で役に立つかもしれないことを書き留めておくように合図し、そのための時間を与える。

避ける：

  1. 記憶に依存するプロセス。例えば：
    • ユーザーに前のステップのコードやカテゴリーを記憶させること。
    • 人を見つけるのが困難な長いメニューシステム。
    • ユーザーが覚えなければならない特定の単語を使用する必要があるシステム。

目標 7: ヘルプとサポートを提供する。

コンテンツのさまざまな理解方法をサポートする。次のような特別なヘルプやサポートを提供する：

  • グラフィック
  • 長い文書の要約
  • 見出しやリンクのアイコン
  • 数字の代わりになるもの

ユーザーがタスクを成功裏に完了できるよう、選択肢を説明してください。

ユーザーが問題に直面したとき、ヘルプを求めたり、フィードバックしたりしやすいようにしてください。ユーザーがフィードバックを送ることが困難な場合、コンテンツを使用することができないかどうかを伝えることができません。ユーザーがいつ問題を経験しているか、あなたは知らないままでいることでしょう。

スマートシティなど、ユーザーデータに依存するアプリケーションもあります。システムを利用できないユーザーからのデータは、データ駆動型システムから欠落する可能性があります。彼らが問題点についてフィードバックすることもできない場合、問題はさらに深刻です。彼らの存在が目に見えなくなり、そのニーズは満たされないからです。

人間によるヘルプを提供する （パターン）

ユーザーニーズ

人間によるヘルプを得る方法を知って、簡単にプロセスを管理したい。

関連するユーザーストーリー： 3.7.1 ヘルプ

やるべきこと

多くの人が、人間によるヘルプを頼りにしている。可能な限り、人間によるヘルプがあり、容易に利用できるようにする。例えば：

  • 各ページ、プロセス中の各ステップで見つけやすい。
  • ユーザーが好むメカニズムで使いやすい。
  • できるだけ少ないステップで済む。例えば：
    • 2つのフィールドがあるフォーム
    • 電子メールアドレス、または
    • 直接担当者につながる電話番号

人間によるヘルプを利用するにあたって、多くの様々なオプションがある音声メニューのように、複雑なメニューシステムを操ることをユーザーに要求してはならない。

サポートスタッフが認知障害や学習障害がある人々を効果的にヘルプして、良いエクスペリエンスを提供できるように、組織の仕組みを整備する。

メリット

ユーザーが何らかの理由で行き詰まったり、混乱したりした場合、人間のヘルプを利用できることが、通常は最も効果的な解決策になります。しかし、実際には、多くのサイトで、複雑なシステムを操作できるユーザーに対してのみ、このオプションが提供されています。

複雑なシステムの例としては、人間の連絡先情報にたどり着くまでに、ユーザーが多くのリンクをたどる必要があるようなプロセスが挙げられます。また、担当者につながるまでに、ユーザーが多くの質問に答えることを要求する電話番号というのもあります。複雑なシステムでは、それを最も必要とする人々が、人間によるヘルプのオプションまでたどり着けないでしょう。このような場合、ユーザーは認知的過負荷に陥り、プロセスを完了するのをやめてしまうかもしれません。また、サービスやその提供者に対して否定的な態度で去ってしまうかもしれません。

例えば、知的障害があるユーザーが、クーポンを利用しようとしたとします。しかし、オンライン購入にクーポンを適用するための手順が見つからず、サポートのための電話番号も見つかりません。そのため、クーポンを利用することができません。

事例

用いる：

  1. 電話番号。理想的には、相互運用可能な音声ユーザーインターフェース（VUI）の仕様を介して、自動的に発信する機能付き。
  2. 例えば、音声メニューシステムで数字「0」を使用して、人間によるヘルプを利用するために、人間または利用可能なスタンダードに直接行く番号。
  3. mailtoプロトコル [[mailto]] を用いて、「to」と「subject」フィールドがあらかじめ記入された電子メールリンク。なお、これは全てのプラットフォームやメールクライアントで動作するわけではないことに注意してください。
  4. ライブチャットやビデオ通話によるヘルプのオプション。なお、ライブヘルプ機能の一部として開く新しいウィンドウは、完全にアクセシブルであり、簡単に閉じることができる必要がある。ライブチャットは、ユーザーのタスクの邪魔にならないようにすること。

避ける：

  1. 人間によるヘルプを利用するために、ユーザーが全て入力しなければならない長い問い合わせフォーム。
  2. ヘルプにたどり着くまでに複数のステップがあるメニュー。
  3. サイト内の数ページからしかリンクされていないページに掲載された連絡先情報。
  4. ヘルプにたどり着くまでに、ユーザーに複数のステップを踏ませること。

複雑な情報やタスクに代替コンテンツを提供する（パターン）

ユーザーニーズ

文章を補足するために、文脈に関連したグラフや写真を提供してほしい。

関連するユーザーストーリー： Help.

やるべきこと

複雑な情報をユーザーが理解するのを補助するコンテンツを提供する。

これには、様々なユーザーグループ向けの、次のような冗長な情報を含む：

  • 理解しやすい言葉で書かれた、長い文書の要約や段階的な情報。
  • 選択肢やデメリットの説明。
  • 表やグラフ。
  • ユーザーにとって馴染みのあるシンボル。
  • よく構成されたビデオコンテンツ。
  • 画像や情報を提供するグラフィック。
  • 数値コンテンツの代替となるもの。

代替または補足的なコンテンツがある場合：

  • ユーザーが最も理解しやすいコンテンツの形式やバージョンを見つけて選択できる、簡単でワンアクションのメカニズムを提供する。
  • 専用のヘルプや代替コンテンツは、メインのコンテンツと明確に区別する。
  • 代替コンテンツと元のコンテンツとの関係を明確にする。
メリット

複雑な情報、長い文書や複雑なデータ形式の使用は、認知的アクセシビリティを必要とするユーザーにとって大きな障壁となる可能性があります。ユーザーは、可能な限り外部からのさらなる支援を必要とすることなく、情報を理解し、記述されたタスクを成功裏に完了することができる必要があります。

コンテンツの主題が複雑な場合もあります。この場合、できるだけ多くのユーザーが、ミスや混乱、支援を必要とせずに理解できるように、慎重な説明、構成および表現が必要になる可能性があります。

グラフや図、表などの情報を提示する方法が、ユーザーにとってより複雑なものになる可能性があります。このような場合、サポートとなる説明やガイド付きの解釈によって、ユーザーが理解する必要のある重要な点を強調することができます。

人によって、理解しやすい情報の種類は異なります。これは、特に認知障害や学習障害がある人に当てはまります。例えば、数字に障害があっても言語に障害がない人もいれば、言語に障害があっても数字を直感的に理解できる人もいます。様々なユーザーをサポートするために、様々な形でヘルプが提供することができます。例えば：

  • 図の説明やヘルプを提供する "脇にある" テキスト。
  • 文章の内容をより分かりやすくするための図やグラフ。
  • タスクが段階的に完了するのを示すビデオクリップ。
  • 複雑ではない補足の表。
  • ホバーするとキーワードを説明するポップアップ（用語集にリンクしている場合もある）
  • プロセス中のステップを示すフローチャート
詳細

数字や複雑な情報に関連するコンテンツに推奨されるテクニック（該当するものを使用してください）：

  • 複雑な情報の理解を助けるために、図表を使用する。
  • 情報の理解を助けるために、表を提供する。
  • 数学の理解がコンテンツを使用する主な要件でない場合は、次のいずれかを使用する：
    • 数字を非数値的な概念で補強する。例えば、とても寒い、冷たい、涼しい、穏やかな、暖かい、熱い、とても熱い。
    • 将来的には、パーソナライゼーション・セマンティクスを使用して、数字以外の概念を追加することもできます。[[Personalization-semantics-1.0]]を参照のこと。
  • 複数のセクションがあるコンテンツでは、次のいずれかを使用する：
    • セマンティクスを有効にして、セクションにシンボルを追加する。
    • 理解を助けるために、見出し、重要な短文や語句に加え、アイコンを追加する。
    • ただし、アイコンを覚えるのが苦手な人もいるので、アイコンと一緒にテキストを併用する。
      • 見やすく、拡大しやすい明確なアイコンやシンボルを使用する。
      • 様々なユーザーが理解できる画像を使用する。
      • 左から右へ読む言語の場合、画像はテキストの左側に配置する。
  • 300文字以上のコンテンツに推奨する手法
    • 一般的な用語や短い文章を使い、理解しやすい要約を提供する。300語未満のコンテンツでは、見出しが要約の役割を果たすことがあります。
    • 見出しを用いて、情報を管理しやすいサイズに分割し、表示される情報を構造化する。これは、特に支援技術の利用者にとって有益である。
    • コンテンツ所有者は、ユーザーの理解を助けるために少なくとも2つのキーワードを特定する。これらのキーワードは、プログラムによって特定可能にして、ユーザーの好みのモダリティで強調されるようにする。
まずはじめに

実世界のタスクを含め、タスクを成功させるために重要で複雑な情報に対して説明的なコンテンツを提供する。

事例

用いる：

  1. 長い文書、複数のセクションがある文書、複雑な情報、および数値の情報については、上記の「詳細」セクションにある手法がある。例えば：
    • 医療処置の説明と成功率の統計は、補足のテキスト、図、グラフを使用することによって増幅される.
    • ビザ申請のための複数のステップは、常に見える全てのステップのフローチャートを追加することによって、使いやすくなる。フローチャートの各ステップにはヘルプへのリンクがあり、現在のステップは明確に強調表示されている。

避ける：

  1. 良い構成と要約がない長い文書。
  2. サポートが提供されていない、複雑な情報や数値の情報。例えば：
    • 長い文章と売上高のデータ表が、その内容に関する重要なポイントの説明なしに表示されている。

アクション、選択肢および選択について、その結果と不利益を明確に示す （パターン）

ユーザーニーズ

選択肢についてのサポートと説明が必要。メリットとデメリットをはっきりさせて、自分がするかもしれない選択の影響を理解したい。

関連するユーザーストーリー： 3.7.5 タスク管理

やるべきこと

ユーザーにアクションや選択肢を提示する場合、それぞれの選択肢の利点、リスク、結果を明確に説明する。例えば：

  • ユーザーが要求したものからの変更。
  • 標準的な製品または提供物からの不利な点。
  • ユーザーの健康や経済面にとってリスクとなる可能性のある機能。
メリット

各アクションや選択オプションの利点と結果を明確に示すことによって、個々がミスを回避することができます。これは、結果を容易に修正できない場合、結果が安全性のリスクにつながる場合、あるいは結果が分からない可能性がある場合、特に重要です。

例えば、ある旅行サイトのユーザーがジュネーブへの旅行を予約しているとします。良いタイミングでオプションを見つけたものの、このチケットは別の都市へのものでした。ユーザーは、そのオプションが自分たちが行きたい目的地のものだと思った。日時を確認するが、スペルしか読めないため、目的地が変更されたことに気づかない。その結果、予約したホテルとは違う場所に連れて行かれてしまい、夜、見知らぬ街に宿無しの状態で到着してしまう。

別の例として、あるユーザーが、ノートパソコンが安く売られているのを見たとします。このとき、ユーザーは長い説明文の中にある「修復済み」という単語に気づきません。そのノートパソコンは、実際にはお得な価格ではありません。

まずはじめに

ユーザーに選択やアクションを求めるときは、ユーザーが認識すべき暗示的または明確ではない結果が潜んでいるかどうかを必ず考慮する。ユーザーインターフェースの中で、そういった結果を明確に示し、ユーザーがそれを認識していることを確認する。

事例

用いる：

  1. 何が含まれ、何が含まれないかを明確にしたリスト。例えば：
    • 航空券を選ぶとき。各オプションの横に、何が含まれるかの明確なリストがある。
  2. あらゆる変更またはリスクに関する警告。例えば：
    • 航空券を選ぶとき、航空券のオプションが異なる目的地のものである場合、またはその他の普通ではないアクションや変更があり、それが望まないものであったり、リスクがあったりする場合は、ユーザーにその変更を確認するよう求める。

避ける：

  1. ユーザーが選択肢を選択する際に、明確でない詳細。
    • 例えば、オンラインメニューの食事オプションには、楽しい名前がついている。食事の内容、サイドメニュー、アレルギー情報、各オプションのカスタマイズ機能などは、プロセスの2ステップ後まで表示されない。顧客は、決定するために各項目で数画面下へ移動しなければならない。
  2. ユーザーに警告することなく、当初のリクエストからアイテムを変更する。
  3. 警告なしにユーザーにリスクを与える。

フォームおよび標準的ではないコントロールに対してヘルプを提供する （パターン）

ユーザーニーズ

馴染みのない操作方法の説明を、自分が使いやすい形（ビデオやテキストなど）で説明してほしい。

関連するユーザーストーリー： 3.7.5 タスク管理

やるべきこと

複雑なフォーム、特に複数のステップ、特殊なインタラクション、標準的ではないコントロール、オートコンプリートに対応していない必須項目がある場合は、ヘルプを提供する。何をすべきかを理解しやすくする例を提示する。

メリット

ユーザーにとって、フォームやそれに関連するタスクは、ウェブサイトで最も複雑な経験となることがよくあります。そのため、ユーザーは簡単に混乱したり、自信がなくなったり、あるいは完全に分からなくなったりすることがあります。追加のヘルプを提供することによって、タスクを完了できるかあきらめるかの違いが生まれます。特に、フォームの一部が複雑であったり、標準的ではないインタラクションを必要とする場合は、そうです。

多くの標準的なフォームコントロールは、自動的にサポートを提供します。例えば、オートコンプリートやパーソナライズセマンティクス [[personalization-semantics-content-1.0]] を用いて、多くのフィールドに自動的に情報を入力することが可能です。そうすれば、ユーザーが入力ミスをすることはありません。

W追加のフィールドや標準的ではないコントロールを要求する場合、多くのユーザーはそれらを使用することが困難になるでしょう。障害があるユーザーの多くは、情報を間違って受け取るか、タスクを完了する方法を見つけることができません。その結果、タスクが完全に放棄されることがよくあります。また、ユーザーが介護者に助けを求めて、フォームに記入したり、コントロールを操作したりするケースもあります。いずれの場合も、障害があるためにタスクを完了することができなかったことになります。

標準の[[HTML]]フォームとコントロールは、最大限のユーザビリティとアクセシビリティを実現するために、仕様書で慎重に定義されています。これらは、特にウェブのインタラクションに慣れているユーザーであれば、通常、理解することができます。しかし、標準的なフォームのふるまいが変更されたり、まったく新しいコントロールが提供されたりすると、ユーザーは困難に直面する可能性があります。新しいふるまいが慎重に設計され、ユーザーテストが行われたと仮定しても、ユーザーがそれらをうまく使うにはヘルプが必要な場合があります。

詳細

追加ヘルプが必要な可能性が高いフォームやコントロールの例：

  • 特定の文字種の入力が必要なパスワード欄。
  • 複雑なインタラクションを伴うアンケート、例えば、直前の回答に応じてのみボタンが表示されるような場合。
  • 日付の入力で、必要な書式があいまいな場合。
  • デートピッカーなどのカスタムコントロールで、一部の日付がグレーアウトしている場合。
事例

用いる：

  1. 標準的なコントロール。
  2. 標準的ではないコントロールと、説明やヘルプのあるフォームフィールド。例えば：
    • 標準的ではないコントロールの横にある、アクセシブルな［ヘルプ］ボタン。
    • 状況依存型ヘルプの標準的なメカニズム。これには、状況依存型ヘルプのためのパーソナライゼーション・セマンティックスの使用を含む。
    • 見つけやすくて、明確であり、曖昧ではない説明文。

避ける：

  1. 説明やヘルプのない非標準的なコントロールや変わったフォームフィールド。例えば：
    • あるフォームでは、スクロールやタブ操作によってセクションを有効にしたり無効にしたりする複雑な仕組みになっているが、ヘルプが提供されていない。

ヘルプを見つけやすく、フィードバックを提供しやすくする（パターン）

ユーザーニーズ

困ったときにはどこからでも簡単にヘルプを利用したり、フィードバックを送ったりしたい。

関連するユーザーストーリー： 3.7.1 ヘルプ

やるべきこと

プロセスのどの時点でも、ユーザーが簡単にヘルプを求めたり、問題を報告できるようにする。例えば：

  • 使いやすい。フィードバック情報とフォームがシンプルで明確であること。(異なるユーザーグループによるユーザーテストを行うことを強く推奨します。）
  • 見つけやすい。ユーザーが行き詰まる可能性のあるどの場所からも利用できること。
  • フォーム、電子メール、チャット、電話サポートなど、好みのコミュニケーション手段を使えること。

ユーザーがフィードバックを提供するための手段は、多くの異なるオプションがある自動音声応答（IVR）のような複雑なメニューシステムをユーザーに使用させるものであってはならない。

メリット

ユーザーが簡単にフィードバックできる手段を提供することによって、ユーザーは問題を共有したり、ヘルプを求めたり、提案したり、肯定的なコメントをしたりすることができるようになります。ユーザーが簡単にフィードバックできない場合、サイトオーナーが気づかないまま、問題が存在し続けることになります。ユーザーがなぜ困っているのかを説明するときに迷子にならないように、ユーザーがプロセス中のどの時点からでもフィードバックできるようにすることが不可欠です。改善のためのアイデアやポジティブなフィードバックも見逃されてしまうでしょう。

例えば、認知障害や学習障害があるユーザーが、ECサイトを使うのに苦労しているとします。どうすればもっと使いやすくなるのか、彼らにはアイデアがあります。そこでフィードバックをしようとして1時間かかってしまったら、やめてしまうでしょう。そのサイトは顧客を失い続けることになります。

詳細

あるべきフィードバックの手段は：

  • 使い方がシンプルである。
  • プロセス中の全ての段階において利用可能である。
  • 提出されたフィードバックに親切に対応するプロセスである。
  • 送信しやすくて、ユーザーに不必要な情報を提供させない。
  • 複雑なメニューシステムに依存しない。

フィードバックを収集するための手段を複数提供することが推奨されます。例えば、Webサイト上では、ライブチャット、電話番号、Webフォーム、フィードバックの電子メールアドレスなど、フィードバックのための4つのオプションすべてを提供することを検討してください。

チャットボットは、フィードバックのプロセスを開始する以外に、この特定のタイプのフィードバックには適していない可能性があることに注意してください。チャットボットは、あなたがリーチしたいエリアに容易にたどり着けない場合には、ものすごくイライラさせられる可能性があります。

事例

用いる：

  1. 各画面に用意されたシンプルなフィードバックのメカニズム。例えば：
    • ある銀行のWebサイトでは、アクセシビリティに大きな問題があり、一部の顧客がオンラインでの請求書払いをブロックされていた。顧客が困ったページには、フィードバックフォームがあった。ある顧客がこの問題を報告することが出来たので、ヘルプデスクの従業員が顧客に連絡して対応し、請求書の支払いを完了できるようヘルプした。また、そのヘルプデスクの従業員は、アクセシビリティの問題をソフトウェアチームに報告した。 ある銀行のWebサイトでは、アクセシビリティに大きな問題があり、一部の顧客がオンラインでの請求書払いをブロックされていた。顧客が困ったページには、フィードバックフォームがあった。ある顧客がこの問題を報告することが出来たので、ヘルプデスクの従業員が顧客に連絡して対応し、請求書の支払いを完了できるようヘルプした。また、そのヘルプデスクの従業員は、アクセシビリティの問題をソフトウェアチームに報告した。
  1. 次のような複数のフィードバックのメカニズム：
    • ウェブチャットまたはウェブコール - ライブチャットまたはビデオ通話を使用してフィードバックを提供するためのオプション。注：ライブチャットまたはビデオ通話機能は、完全にアクセシブルでなければならない。ウェブチャットは、注意散漫にならず、閉じやすいものであるべきである。ユーザビリティをユーザーテストで確認すること。
    • 電話 - フィードバック用の電話番号。理想的には、音声ユーザーインターフェース（VUI）経由で自動的に電話をかける機能が付いていることが望ましい。複雑な音声メニューがないことを確認すること。
    • Webフォーム - 必須項目が3つまでのシンプルな問い合わせフォーム。
    • メール - [[mailto]] プロトコルを使用したEメールリンクで、送信先と件名のフィールドがあらかじめ記入されているもの。なお、全てのプラットフォーム、全てのメールクライアントで動作するわけではないことに注意する。
    • 自動音声応答（IVR） - 応答の最後に、電話の特定の数字を押してフィードバックできる自動オプションを提供する。

避ける：

  1. 複雑なフィードバックのメカニズム。
  2. 誰もが使えるわけではない、1つだけのフィードバック方法。

道案内のヘルプを提供する（パターン）

ユーザーニーズ

道案内やナビゲーションを理解して使用するのをヘルプしてほしい。

関連するユーザーストーリー： Help.

やるべきこと

ユーザーが道案内やナビゲーションシステムを理解し、利用するのに役立つコンテンツを提供する。これには以下が含まれる：

  • 容易に認識できるランドマーク（目印）を提供する。
  • 「タワーの北」のような静的なオブジェクトに関連付けることができる東西南北の方位（一般的またはグローバル）を提供する。
  • 方向や経路の変更など、混乱のもとになる変化を回避できるようにする。
  • ルートから離れるときに方向転換を容易にする。
  • 距離を認識する様々な方法をサポートする。
  • 方向や測定値などの用語のパーソナライズを可能にする。
メリット

認知障害や学習障害がある人は、道案内や道案内のアプリケーションを利用するとき、様々なレベルの困難を経験します。より多くの刺激がある屋内と、より多くの記憶が必要とされる屋外とでは、道案内に関する問題に対処するために必要とされるヘルプは、異なる可能性があります。また、必要なヘルプも、人によって異なります。

道順を理解するには多くの認知機能を必要とします。デザインは、他のデザインパターンでサポートされているものを含め、幅広い認知や学習に関する障害に対応する必要があります。例えば：

  • 記憶
  • 遂行機能
  • 空間的な方向感覚
  • 視覚/空間的な注意、知覚、処理
  • 空間識失調の不安（平衡感覚を喪失した状態）
  • 言語処理
  • 知的
  • 注意力

ユーザーによっては、ステップバイステップの案内など、より詳細なヘルプが必要な場合があります。多くのユーザーは、ルートをたどる前に事前確認する必要があります。その場合、ランドマークは認識と方向付けのヘルプとなり、不安を軽減することができます。ユーザーの好みに合わせて、相対的な方向性を示す代替語や基本方位となる方向を示すことが最も効果的です。例えば、「運転席側」や「東館」などです。また、例えば、「半分の距離を移動したことになります」のように、相対的な距離と絶対的な距離をイメージできるようにするのも役立ちます。

個人の要望は千差万別なので、パーソナライゼーションのメカニズムはとても有用です。例えば、距離に使用する単位です。プラットフォームやその他の技術では、相対的な方向や基本方位、使用できる用語について、パーソナライズのオプションをよく提供しています。例えば、プラットフォームのロケール設定などです。

変更によって、とても混乱することがあります。ユーザーは、渋滞の時間を節約するためにルートを変更する用意があるかもしれませんが、その場合、新しいルートを学ぶためにガソリンスタンドに立ち寄ることになります。わずかな渋滞を避けるためにルートを変更するのは嫌がるかもしれません。ユーザーが自分に合った選択肢を見つけられるよう、手助けしてください。

例えば、外傷性の脳損傷がある人が全地球測位システム（GPS）を使っているとします。彼らは出発前にルートを確認し、曲がり角の写真を見ます。このような準備をすることで、ルートをたどることができるようになるのです。運転中、3分の渋滞を避けるためにルートが変更されたとします。そうすると、もはやルート通りに走れなくなり、道に迷ってしまいます。

事例

用いる：

  1. 常に利用可能な左右を示す第2の方法、または表示をパーソナライズする方法。
  2. ランドマークを利用した道案内。例えば、地元のランドマークの画像を提供したり、追加したりすることで、道案内の際に方向を確認することができる。
  3. 不要な変更を避けるために利用できるオプション。
  4. 追加で利用できるサポート。

避ける：

  1. あまり知られていないポイントも含め、コンパスのポイントを一貫して参照すること（例：N by NE（北東寄りの北））。
  2. 常に想定された場所にいることを前提とした説明。ユーザーは、正しいルートに簡単に戻れない。
  3. ユーザーが望まない、あるいはコントロールできない方法で自動的に変化するコンテンツ。

リマインダーを提供する （パターン）

ユーザーニーズ

カレンダーにリマインダーが欲しい。そうしないと、約束の時間ややるべきことを忘れてしまう。また、次のタスクを完了するためにウェブサイトを再訪問するためのリマインダーが必要なときもある。

関連するユーザーストーリー： 3.7.2 サポート

やるべきこと

Mユーザーが日付や時間に関連するイベントのリマインダーを簡単に設定できるようにする。可能な限り標準的なアプリケーションプログラミングインターフェース（API）を使用すること。

リマインダーはユーザーの要求があった場合のみ設定され、ユーザーはリマインダーの方法をパーソナライズすることができなければならない。

メリット

認知障害や学習障害がある人は、イベントや時間を管理するのが苦手なことがよくあります。(実際、サポートなしでイベントや時間を正しく管理できないことは、障害のグループの一部に関する診断基準になっています)。その結果、会議を欠席したり、特定の日付までにリクエストを提出しなかったり、指定された期間内にフォームを送信しなかったりして、機会を逃します。例えば：

  • ユーザーがカレンダーに情報をコピーするとき、しばしば日付や時間を間違ってコピーしてしまう。
  • ユーザーは、時間ベースの情報を処理し、保持することが困難である。
  • ユーザーは、期限があるイベントの順番を決めるのが難しいかもしれない。
  • タスクを中断しなければならないほど疲れているとき、ユーザーのスキルは低下する。タスクのリスケジュールを希望することがある。

カレンダーAPI（またはタスクマネージャー）を使って、ユーザーが自分のカレンダーに自動的にイベントや締め切りを追加できるようにすることは、有効である。

例えば、認知障害や学習障害があるユーザーが、オンラインで医師の予約を入れたとします。彼らは、その詳細をカレンダーに間違ってコピーしてしまうことがよくあります。しかし、この場合、Webサイトにはカレンダーに予定を追加して、1時間前にリマインダーを設定するオプションがあったとします。ユーザーは、正しい時間に、正しい場所へ、正しい書類を持って来ることができます。

認知に関するアクセシビリティのニーズがあるユーザーにとってのメリットは、予定、締め切り、スケジュールを自分で管理できることです。リマインダーを設定する機能により、期限があるタスクを処理する際の認知的負荷を軽減することができます。時間に依存したアクティビティは、タイムリーに完了するように、ユーザーが監視したり、追跡したりすることが可能です。

ユーザーがタスクを中断されることがないように、タスクの最後にリマインダーを設定するオプションを常に提供してください。

ユーザーのカレンダーがいっぱいになってしまうので、不要なリマインダーを追加しないようにすることが重要です。ユーザーは、カレンダーを全く使うことができなくなってしまう可能性さえあります。リマインダーの数やどんなタイプが自分のニーズに最も合致するかは、ユーザー自身が一番分かっているのです。

詳細

プラットフォームや技術に標準的なメカニズムが存在する場合は、それを使用する必要があります。以下を参照のこと：

  • Web Notification API
  • Using the Notifications API
  • Reminders and Notifications

日時を指定するイベントとは、ある特定の時刻までに完了しなければならないイベントのことです。このようなイベントの時間的制約は、カレンダーの日付と時間、または経過時間の合計によって定義されます。

考慮されうる変数には、以下のものがあります：

  • 時間 - 論理的な時間。
  • 場所 - 適切な場所にいるときに表示される。
  • コンテキスト - コンピュータかモバイルか、特定のサイトか、など。
事例

用いる：

  1. ユーザーがカレンダーにイベントを追加し、リマインダーを設定するためのオプション。例えば：
    • ヘルスケアサイトでは、地域の医療予約を設定することができる。一旦予約が設定されると、ユーザーはそれをカレンダーに（自動的に）追加し、3時間前にリマインダーを設定するオプションが与えられる。また、リマインダーの追加や編集も可能。

避ける：

  1. ユーザーのカレンダーに追加された、ユーザーが参加したくないイベント。
  2. ユーザーが設定したばかりのイベントや予定を、自動的にカレンダーに追加することができない。例えば：
    • あるヘルスケアサイトでは、地域の診療予約を設定することができる。ユーザーには、それを自動的にカレンダーに追加したり、リマインダーを設定したりするオプションが与えられていない。

パーソナライゼーションは、ユーザーが一連の代替手段の中から好みの、使い慣れたオプションを選択することを可能にします。

可能な限り、パーソナライゼーションと簡素化をサポートしましょう。アドオンや拡張機能を無効にしないでください。

コンテンツがいつ移動または変更されるかをユーザーが制御できるようにする（パターン）

ユーザーニーズ

どこに何があるかを知りたい。コントロールとコンテンツは、私が使用しているときには予期せずに移動しない。

関連するユーザーストーリー： 3.8.1 適応

やるべきこと

コンテキスト、機能、設定、ルート、方向などの変更は、ユーザーの要求によってのみ開始されるようにするか、そのような変更を簡単にオフにできるメカニズムを利用できるようにする。また、以前のコンテキスト、機能、設定、ルート、および方向に戻るための簡単に利用できるメカニズムを提供すること。

メリット

コンテンツ、設定、機能などがユーザーの意思によらず不意に変更されると、認知障害や学習障害があるユーザーにとって重大な問題になりえます。これらの予期せぬ変更は、ユーザーインターフェース（メニュー、ボタン、デザインコンポーネントなど）の理解または使用において、集中力の低下、不安、混乱を引き起こす可能性があるからです。例としては、以下のようなものが挙げられますが、これらに限定されるものではありません。

  • 新しいウィンドウやポップアップが自動的に起動すること。
  • （フォームを送信することが分かる平易な言葉を用いて）明確にラベル付けされたボタン以外の仕組みでフォームを送信すること。
  • 新しいコンテンツや機能が開くこと。
  • オプション選択時の予期せぬ変更。
  • GPS（全地球測位システム）による自動リルート。
  • GPSで地図の向きを変更すること。

例えば、あるユーザーは方向音痴で、自分の左右が分からないかもしれません。自分が何をしているかをおおよそ理解し、GPSを手がかりにして自分のコンテキストでGPSが示す道順を把握しやすくするために、GPSを使う前にルートを調べておくことがあります。GPSは、わずかな交通渋滞のために自動的にルートを変更します。そうすると、ユーザーは完全に道に迷って、方向感覚を失ってしまい、アプリケーションを使用できなくなる可能性があります。

別の例として、あるユーザーがビデオを見ていて、「いいね！」を押そうとしたとします。ボタンを押そうとすると、コントロールが移動し、「いいね！」を押せずに別の動画を読み込んでしまいました。ユーザーは自分のコンテンツを見失いたくないので、「いいね！」を押さなくなってしまいます。その結果、ユーザーの好みが反映さなくなります。

コンテンツが変わるタイミングをユーザーがコントロールできるようにすれば、認知障害や学習障害があるユーザーが、ウェブサイトやアプリケーションのふるまいをよりコントロールしやすくなります。これにより、コンテンツを利用したり、タスクを完了したりできるようにするための選択をする機会をユーザーに提供できます。

詳細

例外：変更が、アクティビティにとって不可欠な部分である場合（例：ゲーム）。

ルート：GPSのルートなど、道案内や流れ。

オリエンテーション：地図の道案内など、全体像や表示です。

簡単に利用できる（または、簡単に利用できるモードもしくは設定）というのは、以下のうち1つ以上が当てはまる場合です：

  • できるだけ広い範囲で一度に設定できる（オペレーティングシステム（OS）の規格を使用する。利用可能な場合は、[[ISO9241-112]]や[[GPII]]など）。
  • 対象となるウェブページの範囲を保存したり、設定を変更したりすることが可能なオプションがある。
  • 必要な各画面から到達可能であり、そのパスとコントロールは本ドキュメントの全てに適合している。
事例

用いる：

  1. コンテンツの変更に対するユーザーによる制御。例えば：
    • ユーザーは「一定以上の時間を短縮できるなら、ルートを変更する」という設定にすることが可能。ユーザーは、5分短縮できるなら追加で何回曲がってもよい、などの情報をさらに追加できる。GPSは時間を短縮できる新しいルートを見つけたとき、追加される曲がり角の数と短縮できる時間を含めて、ユーザーに変更内容を通知する。GPSは、ユーザーがルートを変更したいかをユーザーに確認するか、またはGPSがルートを変更した場合、ユーザーはワンタッチまたはコマンドで元のルートに戻ることができる。

避ける：

  1. ユーザーが制御できない変化をするコンテンツ。

  • 数字を単語に、単語を数字に変更。
  • 行間、文節、フレーズ、チャンクに空白を追加。
  • 音声認識など、コンテンツ入力の代替方法。
  • 画像を追加。

しかし、Webサイトが拡張機能やAPIを動作させないことがあります。その結果、ユーザーはWebサイトを利用することができなくなります。

こういったアドオンやAPIがサポートされていない場合、支援技術として使用されるアドオンのすべての機能のサポートを提供する必要があります。

例えば、外傷性の脳損傷があるユーザーは、次のような詳細を記憶する能力に影響を及ぼす遂行機能や記憶障害があることがあります：

  • Web of Things（WoT）インターフェイス上のアイコンやシンボル
  • ユーザー名とパスワード
  • 頭字語の意味
  • 電話番号
  • 一般的でない言葉の意味

コンテンツを簡略化し、サポートを提供するアドオン（頭字語の長い形式、ポップアップ辞書など）の使用をサポートすることで、ユーザーはほとんどのコンテンツを理解することができます。

パスワード管理ツールをサポートすることにより、ユーザーは問題なくログインして、安全なサイトから締め出されることを回避できます。

非機密情報の保存とオートコンプリートにより、フォームへの入力が容易になります。例えば、電話番号や住所などの一般的な情報が該当します。また、入力ミスを防ぐこともできます。このような情報を記憶から正確に呼び出したり、コピー＆ペーストしたりする必要がなくなります。こういったタスクによって、ユーザーはフォームをうまく利用できないことが多いのです。

テキストコンテンツに圧倒されてしまうとき、必要なコンテンツを見つけるのに役立つ、彼らがよく知っているシンボルを挿入する拡張機能があります。

オプションが多すぎると、IoTデバイスの操作が複雑になってしまうことがあります。追加オプションは無視しやすくすべきです。また、たくさん読まなくても、それが追加オプションであることやスキップする方法を理解できるようにすべきです。

例えば、メーターのデフォルトの「読み取り値」が「1」ではなく「2」に設定されているなど、Internet of Things （IoT）のインターフェースはユーザーを混乱させることがあります。その場合、ユーザーは "1 "にリセットする必要があるでしょう。

どんなソリューションの提案においても、IoTとのインタラクションなどの操作タスクを可能な限り透明化して、ユーザーがコンテンツとの関連付けなどの機能面に注意を向けられるようにすることが重要です。

詳細

認知障害や学習障害がある人は、支援技術としてアドオンをよく使用しています。アドオンや類似のツールが期待通りに動作することは不可欠ですが、以下の場合は例外です：

  1. セキュリティまたは安全性の要件により、これらのAPIを無効にする必要がある場合。この場合、関連するフィールドに対してのみ無効にする必要があります。
  2. アドオンが、評価やテスト用アプリケーションなど、サイトの主要な機能を壊している場合。

アドオンがコードによって自動的に無効にされる場合、アドオンが追加する機能を提供する負担が制作者にかかります。

まずはじめに

認知障害や学習障害がある人をサポートするAPIや拡張機能でコンテンツを利用できるようにする。

コンテンツに適したいくつかの API を使用することによって検証されるテストを行う。例えば：

  • スペルチェッカーやパスワード保存のアプリや拡張機能でテストする。
  • メニュー項目に追加する拡張機能（デスクトップの右クリックオプション）でテストする。
  • 簡略化またはパーソナライズを可能にし、認知障害や学習障害がある人のために設計されたツールバーでテストする。
事例

用いる：

  1. ブラウザの拡張機能や機能、パーソナライゼーション・ツールバーへの対応。ユーザーが、パーソナライゼーションツールバーから設定を適用して、ページのユーザビリティを向上できるようにする。
  2. 右クリックにオプションを追加する拡張機能が、期待通りに動作する。
  3. オペレーティングシステムやユーザーエージェントのユーザー設定によって、ページの書式変更や簡略化を可能にする。

避ける：

  1. ブラウザの拡張機能、設定や機能、パーソナライズツールバーが機能しないようにする設計やコード。例えば：
    • パスワード保存アプリケーションが動作しない。
    • 注意力散漫を解消する拡張機能が動作しない。
    • スペルチェックの拡張機能が、右クリックメニューにオプションを追加しない、または、ユーザーが犯した間違いに下線を引かない。
    • 簡略化ツールバーで正しい記号を追加できない。
    • ページの書式設定や余分な空白の追加ができない。
    • ユーザーが好みのフォントを使用できない。

Support Simplification (Pattern)

ユーザーニーズ

I need less content without extra options and features as I cannot function at all when there is too much cognitive overload.

関連するユーザーストーリー： Adapt.

やるべきこと

Support simplification of your content. Often this includes allowing the user to:

  • Remove or hide features that most users do not use or that are not essential.
  • Get less text or more simple text.
  • Select the content format or version that is easiest for them to understand, or
  • Find the extra features when wanted.
メリット

A user who has difficulty reading or using web content can be easily overwhelmed with too much information on a web page. They need to simplify the page to include the critical information that they need and not spend all their energy reading and understanding other content and features. This is also true for users who are easily distracted.

For example, an email program has lots of features and formatting options when drafting an email. This makes it too complex for a lot of people. With personalization the user can have a simple option with only send and cancel options. There is a “to” and subject line but no cc or bcc options. In this setting there is a clear heading (write an email) and they have icons that the user understands.

詳細

Note that:

  • Typically, a simple application has 3 to 6 functions.
  • Make sure it is easy to get back to the full featured version.
  • You can meet this design pattern by:
    • supporting simplified versions from the browser,
    • using of data-simplification on regions and controls,
    • using other attributes in personalization semantics (see [[personalization-semantics-1.0]]),
    • adding a simplification toolbar, or
    • providing an alternative version.
まずはじめに

Add data-simplification="critical" on content that is in any critical user testing paths.

事例

用いる：

  1. A simplified “reading” view that is available and easy to close.
  2. Applications that have 3 large features. Other features are in the footer or under a “more” option.
  3. A simplified version of the application is available.

避ける：

  1. Simplified modes have unnecessary extra content or are not supported.
  2. Applications with lots of features and cannot be simplified. For example:
    • A busy email program has many control bars and features such as tagging, group tagging, start a new thread, etc. There is no easy way to simplify the page.

Support a Personalized and Familiar Interface (Pattern)

ユーザーニーズ

I need (a version of) a familiar interface, that I recognize and know what will happen.

関連するユーザーストーリー： Adapt.

やるべきこと

Provide users with a way to personalize their interface to make it familiar.

This can be done by:

  1. Allowing user preferences on presentation such as font style, font size, line heights, margins, and contrast. (Note: The default version should still be readable and use clear fonts.)
  2. Allowing a rollback to a previous interface that the user is familiar with and knows how to use.
  3. Adding semantics on controls, links, and symbols that allow the user to control the experience. For example:
    • HTML5 autocomplete on common fields,
    • adding a toolbar that adds personalized images,
    • use attributes in [[personalization-semantics-1.0]].

Ensure the user knows the personalization options and can easily configure them. Clear instructions can help.

メリット

Personalization changes the interface to meet the needs of the user.

Having familiar terms and symbols is key to many users being able to use the web. However, what is familiar for one user may be unfamiliar to another requiring them to learn new symbols. Adding semantics allows symbols and support to be added by an extension or browser that is familiar to the individual user.

A stronger example is people using AAC. These users usually only learn one symbol set. They cannot easily communicate with other people using AAC in a written format or may struggle to understand different symbols used in different applications. When using personalization, such as [[personalization-semantics-1.0]] the user agents can load the symbols that are understandable by the individual user. The user can also access the Web and other applications.

Other support includes autocomplete and extensions that help the user fill out forms and understand the content. Many users with memory or executive function impairments cannot fill in forms without asking someone to help copy over information or check their work. Autocomplete allows many more users to manage forms by themselves.

まずはじめに
  • Use the HTML autocomplete tags on all common fields.
  • Add a toolbar that adds personalized images.
  • Add the semantics that can work with a toolbar for personalized images.
事例

用いる：

  1. [[HTML5]] autocomplete tags on all common fields.
  2. True text to support browser preferences for styles.
  3. A toolbar that adds personalized images.
  4. Semantics that can work with a toolbar for personalized images or [[personalization-semantics-1.0]].

避ける：

  1. Forms that do not support [[HTML5]] autocomplete.
  2. Default fonts that are not clear or readable such as a cartoon font or gothic.
  3. Pages that cannot easily be personalized.
